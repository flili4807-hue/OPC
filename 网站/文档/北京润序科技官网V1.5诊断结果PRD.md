## 1. 背景

当前项目成功率自评问卷已经支持 24 个问题、8 个维度、0-5 分选择、总分计算、CMMI 成熟度映射和基础治理建议。

但当前“生成诊断摘要并预约 / 带评测结果预约诊断”链路仍存在明显断点：

1. 用户只看到总分，无法快速理解自己具体弱在哪些维度。
2. 项目经理无法基于一个总分判断问题根因，仍需要重新访谈。
3. 当前没有后台，无法自动获取用户问卷答案和维度数据。
4. 没有可复制的结构化结果，用户也无法方便地把诊断信息带给项目经理。

因此 V1.5 需要先在不建设后端的前提下，完成“前端诊断结果闭环”。

## 2. 产品目标

### 2.1 业务目标

1. 把问卷从“打分工具”升级为“售前诊断工具”。
2. 让用户一眼看懂项目治理短板。
3. 让项目经理拿到足够的结构化上下文，能够快速进入诊断和解决方案设计。
4. 在没有后台的 V1.5 阶段，通过“复制诊断摘要”解决结果传递问题。
5. 为 V2 后台保存问卷结果、飞书通知、诊断报告生成预留数据结构。

### 2.2 用户目标

用户完成或调整问卷后，应能快速知道：

1. 我的项目整体成功率是多少。
2. 我的 8 个治理维度分别表现如何。
3. 最短板的 3 个维度是什么。
4. 每个短板维度应该优先怎么治理。
5. 如何把诊断结果发给润序科技进行项目诊断。

### 2.3 项目经理目标

项目经理收到用户复制的诊断摘要后，应能快速判断：

1. 项目主要风险集中在哪些维度。
2. 是否属于目标范围不清、组织职责不清、需求变更失控、计划风险不足、质量门禁不足、数据度量不足、工具流程不足或复盘机制不足。
3. 首次诊断会应该重点问哪些问题。
4. 后续解决方案应该从哪些治理动作切入。

## 3. 本期范围

### 3.1 V1.5 包含

1. 8 维度得分计算。
2. 8 维度雷达图。
3. 8 维度列表展示。
4. 短板 Top 3 自动识别。
5. 每个短板维度对应治理建议。
6. 诊断摘要自动生成。
7. 一键复制诊断摘要。
8. CTA 文案调整为“生成诊断摘要并预约”。
9. 不保存数据、不上传数据、不建设后台。
10. 不新增接口、不接入统计埋点、不向外部服务发送问卷结果。

### 3.2 V1.5 不包含

1. 后台管理系统。
2. 问卷结果数据库保存。
3. 用户登录。
4. 自动发送飞书通知。
5. 邮件自动发送。
6. PDF 报告生成。
7. 项目经理后台查看结果。
8. 多人协作诊断。
9. 联系方式表单提交。
10. 用户评测结果自动传递给润序科技。

## 4. 用户路径

```text
用户进入问卷页
→ 默认所有题目为 0 分
→ 用户根据实际情况调整 0-5 分
→ 系统实时计算总分和 8 维度得分
→ 雷达图实时更新
→ 系统识别短板 Top 3
→ 系统生成维度建议
→ 用户点击“复制诊断摘要”
→ 用户点击“生成诊断摘要并预约”跳转到联系区
→ 用户通过飞书/邮箱主动发送摘要给项目经理
→ 项目经理根据摘要安排诊断和解决方案设计
```

分支说明：

1. 如果用户未调整任何题目，默认 0 分也允许复制摘要和预约诊断。
2. 如果复制失败，页面展示可手动复制的摘要文本区域，用户仍可继续点击预约诊断。
3. 页面刷新后不保留问卷结果，恢复默认 0 分状态。

## 5. 评分模型

### 5.1 问卷结构

| 维度 | 题目编号 | 题数 |
|---|---|---:|
| A. 目标与范围 | A1-A3 | 3 |
| B. 组织与职责 | B1-B3 | 3 |
| C. 需求与变更 | C1-C3 | 3 |
| D. 计划与风险 | D1-D3 | 3 |
| E. 质量与测试 | E1-E3 | 3 |
| F. 数据与效能 | F1-F3 | 3 |
| G. 工具链与流程 | G1-G3 | 3 |
| H. 复盘与持续改进 | H1-H3 | 3 |

### 5.2 单题分值

| 分值 | 含义 |
|---:|---|
| 0 | 没有 / 未建立 / 不适用但未处理 |
| 1 | 极弱，主要依赖个人经验 |
| 2 | 初步存在，但不稳定 |
| 3 | 基本可用，但缺少统一机制 |
| 4 | 较成熟，有明确流程和责任 |
| 5 | 成熟稳定，可量化、可复用、可持续改进 |

### 5.3 总分计算

```text
总分 = round(全部 24 题分值总和 / 120 × 100)
```

### 5.4 维度得分计算

```text
维度得分 = round(该维度 3 题分值总和 / 15 × 100)
```

### 5.5 总分等级

| 总分 | 判断 | CMMI 参考映射 | 总体建议 |
|---:|---|---|---|
| 85-100 | 高成功概率 | L4-L5 | 强化数据驱动和组织级最佳实践沉淀 |
| 70-84 | 可控推进 | L3-L4 | 加强风险前置、发布门禁和复盘闭环 |
| 55-69 | 需要治理 | L2-L3 | 优先补齐计划、依赖、风险和质量机制 |
| 40-54 | 高风险 | L1-L2 | 先做项目治理重构再加速交付 |
| 0-39 | 极高风险 | L1 | 当前更适合先做基础治理梳理，优先完成目标、范围、组织和质量体系梳理 |

展示口径：

1. 页面和摘要中统一使用 `参考映射：CMMI Lx`，不得表达为官方评估、认证或评级结果。
2. 风险等级可以展示，但必须配套建设性建议，避免只输出负面结论。

## 6. 功能需求

### 6.1 雷达图

#### 目标

用图形化方式展示 8 个治理维度的相对强弱，让用户和项目经理快速识别治理短板。

#### 展示规则

1. 雷达图包含 8 个轴，对应 8 个维度。
2. 每个轴分值范围为 0-100。
3. 页面初始状态全部为 0。
4. 初始 0 分状态下，雷达图仍需展示坐标网格、8 个维度标签、中心点和说明文案：`当前各维度均为 0 分`。
5. 用户调整任一题后，雷达图实时更新。
6. 雷达图用于快速判断维度强弱，8 维度列表用于读取准确分数。
7. Canvas 实现需适配 `devicePixelRatio`，并在窗口尺寸变化时重绘。
8. 图形应与当前深海蓝科技风格一致。
9. 不引入第三方图表库，优先使用 Canvas 原生绘制；如使用 SVG，需满足同等响应式和可读性要求。

#### 验收标准

| 前置条件 | 操作 | 预期结果 |
|---|---|---|
| 用户打开问卷页 | 无操作 | 雷达图 8 个维度均为 0 |
| 用户打开问卷页 | 无操作 | 雷达图显示网格、中心点、8 个维度标签和“当前各维度均为 0 分”说明 |
| 用户调整 A 维度题目 | 选择 A1-A3 分值 | 雷达图 A 维度实时变化 |
| 用户调整多个维度 | 修改不同分组分值 | 雷达图显示各维度相对差异 |
| 用户调整浏览器宽度 | 缩放或旋转设备 | 雷达图重绘，不模糊、不溢出 |

### 6.2 8 维度得分列表

#### 目标

除雷达图外，以列表方式提供明确数值，避免只看图不够准确。

#### 展示字段

每个维度展示：

1. 维度名称。
2. 得分，0-100。
3. 成熟度状态。

状态规则：

| 维度分 | 状态 |
|---:|---|
| 85-100 | 优势维度 |
| 70-84 | 稳定维度 |
| 55-69 | 待加强 |
| 40-54 | 风险维度 |
| 0-39 | 核心短板 |

#### 验收标准

| 前置条件 | 操作 | 预期结果 |
|---|---|---|
| 用户打开问卷页 | 无操作 | 8 个维度均显示 0 分和核心短板 |
| 用户调整某维度 | 修改该维度题目 | 对应维度分值和状态实时更新 |

### 6.3 短板 Top 3

#### 目标

自动识别最需要优先治理的 3 个维度，帮助用户和项目经理快速聚焦。

#### 排序规则

1. 按维度得分从低到高排序。
2. 得分相同时，按问卷维度顺序 A-H 排序。
3. 取前 3 个维度作为短板 Top 3。

#### 展示字段

1. 维度名称。
2. 维度得分。
3. 对应治理建议。

#### 验收标准

| 前置条件 | 操作 | 预期结果 |
|---|---|---|
| 用户打开问卷页 | 无操作 | Top 3 为 A、B、C，分值均为 0 |
| 用户提高 A、B、C 分值 | 修改对应题目 | Top 3 自动更新为当前最低的三个维度 |

### 6.4 维度治理建议

#### 目标

让诊断结果不只是分数，而是能直接辅助项目经理形成诊断和解决方案。

#### 建议内容

| 维度 | 指导建议 |
|---|---|
| 目标与范围 | 优先明确业务目标、用户目标、成功指标、本期范围和不做事项。 |
| 组织与职责 | 建立项目 Owner、RACI 职责矩阵、跨团队依赖响应机制。 |
| 需求与变更 | 建立 PRD/原型/技术方案准入，明确变更评估、审批和排期影响。 |
| 计划与风险 | 建立里程碑、关键路径、风险台账、决策事项和回滚预案。 |
| 质量与测试 | 补齐测试计划、缺陷等级、准入准出、发布门禁和 Go/No-Go 机制。 |
| 数据与效能 | 建立进度、质量、效率、风险指标和可视化效能看板。 |
| 工具链与流程 | 将需求、任务、缺陷、风险、版本在线化管理，建立版本和 CI/CD 机制。 |
| 复盘与持续改进 | 建立版本复盘、问题闭环、SOP 模板和知识库沉淀机制。 |

### 6.5 诊断摘要生成

#### 目标

在无后台的 V1.5 阶段，生成可复制的结构化文本，用户可通过飞书或邮箱发送给项目经理。

#### 摘要内容

诊断摘要必须包含：

1. 总分。
2. CMMI 参考映射。
3. 8 维度得分。
4. 短板 Top 3。
5. Top 3 对应治理建议。
6. 免责声明：本工具用于项目管理诊断沟通，不等同于官方 CMMI 评估或认证。

诊断摘要不包含：

1. 24 道题逐题答案。
2. 用户身份信息。
3. 联系方式。
4. 任何系统生成的用户 ID。

#### 示例格式

```text
【Runxu Tech 项目成功率自评摘要】
总分：42
等级：高风险 · 参考映射：CMMI L1-L2

8 维度得分：
- 目标与范围：40
- 组织与职责：53
- 需求与变更：33
...

短板 Top 3：
1. 需求与变更 33：建立 PRD/原型/技术方案准入...
2. 目标与范围 40：优先明确业务目标...
3. 质量与测试 47：补齐测试计划...

说明：本工具用于项目管理诊断沟通，不等同于官方 CMMI 评估或认证，不是完整解决方案。
```

### 6.6 复制诊断摘要

#### 目标

降低用户把结果发给项目经理的成本。

#### 交互规则

1. 点击“复制诊断摘要”后，将摘要文本写入剪贴板。
2. 成功后按钮文案变为“已复制”，保持 1.5-2 秒后恢复为“复制诊断摘要”。
3. 如果复制失败，展示可手动复制的摘要文本区域。
4. 复制不上传数据，不触发外部网络请求。
5. Clipboard API 不可用、浏览器权限拒绝、非 HTTPS 环境限制等场景，均视为复制失败，使用手动复制降级方案。
6. 默认 0 分状态下也允许复制摘要。

#### 验收标准

| 前置条件 | 操作 | 预期结果 |
|---|---|---|
| 用户完成评分调整 | 点击复制摘要 | 摘要写入剪贴板 |
| 用户未调整任何题目 | 点击复制摘要 | 摘要写入剪贴板，摘要显示总分 0、极高风险、参考映射 CMMI L1 |
| 浏览器禁止剪贴板 | 点击复制摘要 | 页面展示可手动复制文本 |
| Clipboard API 不可用 | 点击复制摘要 | 页面展示可手动复制文本 |
| 复制成功 | 点击复制摘要 | 按钮显示“已复制”1.5-2 秒后恢复 |

### 6.7 CTA 调整

#### 当前问题

“带评测结果预约诊断”表达不准确，因为页面未真正把结果传递给后台或项目经理。

#### V1.5 CTA

主 CTA 改为：

`生成诊断摘要并预约`

辅助动作：

`复制诊断摘要`

#### 交互

1. 用户先点击“复制诊断摘要”。
2. 再点击“生成诊断摘要并预约”跳转到联系区：`index.html#contact`。
3. 用户通过飞书/邮箱把摘要发给项目经理。
4. 联系区需提示：`请将诊断摘要通过飞书或邮箱发送给我们，便于快速判断项目治理重点。`
5. 如果用户未复制摘要直接点击预约，也允许跳转联系区，但页面需在 CTA 附近提示：`建议先复制诊断摘要，再预约诊断。`

## 7. 数据与隐私

### 7.1 V1.5 数据策略

1. 所有计算在浏览器本地完成。
2. 不保存用户答案。
3. 不上传用户答案。
4. 不生成用户 ID。
5. 不调用后端接口。
6. 不自动发送飞书或邮件。
7. 不新增提交接口。
8. 不通过表单自动发送问卷结果。
9. 不接入统计脚本。
10. 不向任何第三方服务发送评测数据。
11. 诊断摘要仅在用户主动点击复制后写入本地剪贴板。

### 7.2 后续 V2 预留

如果后续要让后台获取用户数据，需要另开 V2 后端方案，至少包含：

1. 用户授权。
2. 隐私政策。
3. 结果 ID。
4. 结果保存接口。
5. 项目经理查看入口。
6. 飞书/邮件通知。
7. 数据删除机制。
8. 数据保存周期。
9. 结果对象字段：总分、CMMI 参考映射、8 维度分、Top 3、逐题答案、联系方式、授权状态、创建时间。

## 8. 非功能要求

1. 不引入第三方图表库。
2. 24 个问题中任一分值变化后，总分、维度分、Top 3 和雷达图更新不得出现明显卡顿。
3. Canvas 需适配高清屏，避免模糊。
4. 移动端雷达图、列表和下拉框不应溢出。
5. 复制摘要失败时必须有降级方案。
6. 页面刷新后不保留问卷结果。
7. 不产生任何外部数据传输。
8. 375px、768px、1440px 三档宽度下页面不横向滚动。
9. 不新增超过现有静态官网规模的大型依赖。

## 9. 埋点要求

V1.5 不接入统计埋点。

如后续接入统计，建议事件：

| 事件 | 触发 |
|---|---|
| diagnostic_score_changed | 用户调整任一题目 |
| diagnostic_summary_copy | 用户点击复制摘要 |
| diagnostic_contact_click | 用户点击生成诊断摘要并预约 |

接入前需补充隐私和统计说明。

V1.5 硬约束：

1. 不接入任何统计脚本。
2. 不发送任何埋点事件。
3. 不记录用户评分行为。

## 11. 需求风险

| 风险 | 影响 | 应对 |
|---|---|---|
| 无后台导致项目经理无法自动收到结果 | 仍需用户主动发送摘要 | V1.5 提供复制摘要；V2 再做后台 |
| 雷达图在移动端可读性不足 | 用户看不清维度 | 移动端配合维度列表 |
| 用户误以为是官方 CMMI 评估 | 合规风险 | 保留免责声明 |
| 摘要过长 | 用户不愿复制发送 | 摘要控制为结构化短文本 |
| 复制摘要失败 | 用户无法顺畅传递结果 | 展示手动复制文本区 |
| 页面直接跳转联系区但用户未复制摘要 | 项目经理仍拿不到结果 | CTA 附近提示先复制摘要，联系区提示发送摘要 |
| 雷达图 0 分状态像异常 | 用户误解页面未加载 | 展示网格、中心点、维度标签和 0 分说明 |

## 13. 前端实现约束

### 13.1 数据结构建议

页面显示、雷达图、Top 3 和诊断摘要必须共用同一套计算结果，避免展示内容和复制内容不一致。

建议维度对象结构：

```js
{
  key: 'A',
  name: '目标与范围',
  short: '目标',
  fields: ['A1', 'A2', 'A3'],
  advice: '优先明确业务目标、用户目标、成功指标、本期范围和不做事项。',
  score: 0,
  status: '核心短板'
}
```

### 13.2 页面元素建议 ID

| 元素 | 建议 ID |
|---|---|
| 雷达图 | `radarChart` |
| 8 维度列表 | `dimensionList` |
| Top 3 短板 | `weaknessList` |
| 复制按钮 | `copySummary` |
| 手动复制文本框 | `manualSummary` |
| 预约 CTA | `diagnosticContact` |

### 13.3 移动端模块顺序

```text
顶部导航

复制摘要 / 预约诊断

8 维度得分列表
手动复制文本区
```
