## 5. 分角色详细评审

### 5.1 产品负责人

结论：通过。

认可点：

1. V1.5 解决了当前转化链路的真实问题：用户和项目经理都不能只看总分。
2. 范围控制合理，先做前端诊断结果闭环，不在 V1.5 引入后台复杂度。
3. 评分模型、Top 3、摘要复制都能支撑售前诊断。

问题与建议：

| 优先级 | 问题 | 修改要求 |
|---|---|---|
| P1 | “复制摘要”和“预约诊断”的先后关系还不够强 | 在 PRD 的 CTA 章节增加说明：用户应先复制摘要，再跳转联系区发送摘要。 |
| P1 | CTA 跳转目标不够明确 | 明确跳转地址为 `index.html#contact`。 |
| P2 | 完全 0 分也可预约，但咨询质量可能低 | 明确 0 分摘要也可复制，并在摘要中提示“建议先做基础治理梳理”。 |

### 5.2 UI/UX 设计

结论：有条件通过。

认可点：

1. PRD 对结果页的信息结构提出了明确要求。
2. 雷达图 + 维度列表 + Top 3 的组合合理，兼顾直观性和准确性。
3. 移动端不溢出、维度列表兜底的方向正确。

问题与建议：

| 优先级 | 问题 | 修改要求 |
|---|---|---|
| P1 | 雷达图初始 0 分状态只写“全部为 0”，没有写视觉表现 | 增加规则：0 分时仍显示网格、中心点、8 个维度标签和说明文案。 |
| P1 | 8 维度列表和雷达图的主次关系未明确 | 明确雷达图用于趋势判断，维度列表用于准确分数阅读。 |
| P2 | 移动端布局只要求不溢出，缺少模块排序 | 增加移动端顺序：总分、雷达图、Top 3、按钮、问卷、维度列表。 |

### 5.3 前端研发

结论：有条件通过。

认可点：

1. 纯前端计算、无接口依赖，开发路径清晰。
2. 不引入第三方图表库，有利于保持静态官网轻量。
3. 评分公式和排序规则已经明确。

问题与建议：

| 优先级 | 问题 | 修改要求 |
|---|---|---|
| P1 | Canvas / SVG 方案没有明确最终实现口径 | 建议 PRD 明确优先 Canvas，并要求 `devicePixelRatio` 高清适配。 |
| P1 | 窗口尺寸变化未定义 | 增加 resize 后雷达图重绘要求。 |
| P1 | 复制成功反馈只写“短暂”，不可验收 | 明确 `已复制` 保持 1.5-2 秒后恢复。 |
| P2 | DOM 元素命名未在 PRD 中定义 | 增加建议 ID：`radarChart`、`dimensionList`、`weaknessList`、`copySummary`、`manualSummary`。 |

### 5.4 后端研发

结论：通过。

认可点：

1. V1.5 明确不保存、不上传、不建设后台，后端无开发工作量。
2. V2 预留方向完整，包括结果 ID、保存接口、项目经理查看入口、飞书通知。

问题与建议：

| 优先级 | 问题 | 修改要求 |
|---|---|---|
| P1 | PRD 中需要进一步强调 V1.5 不新增任何接口 | 在数据与隐私章节增加：不得新增提交接口、不得通过表单自动发送结果。 |
| P2 | V2 预留字段还不完整 | 后续 V2 单独定义结果对象：总分、维度分、Top 3、逐题答案、联系方式、授权状态、创建时间。 |

### 5.5 测试

结论：有条件通过。

认可点：

1. 功能验收、边界验收和合规验收已经分层。
2. 0 分、满分、单维高分、同分排序都有验收口径。

问题与建议：

| 优先级 | 问题 | 修改要求 |
|---|---|---|
| P1 | 复制失败只写“浏览器禁止剪贴板”，覆盖不足 | 增加 Clipboard API 不可用、权限拒绝、非 HTTPS/本地环境限制等失败场景。 |
| P1 | 移动端只验 375px 不够 | 增加 375px、768px、1440px 三档验收。 |
| P2 | Top 3 同分排序用例不够具体 | 增加用例：A/B/C 提高到高分后，Top 3 应顺延为 D/E/F。 |
| P2 | 页面刷新后不保留结果需验收 | 增加刷新后恢复默认 0 分的用例。 |

### 5.6 内容 / 品牌负责人

结论：有条件通过。

认可点：

1. 摘要结构清晰，适合用户转发给项目经理。
2. 免责声明必要，降低 CMMI 误解风险。
3. 治理建议体现润序科技的专业服务能力。

问题与建议：

| 优先级 | 问题 | 修改要求 |
|---|---|---|
| P1 | “CMMI 映射”容易被理解为正式评估 | 统一改为“参考映射：CMMI Lx”。 |
| P1 | “极高风险”文案可能过强 | 保留风险等级，但增加建设性建议，例如“当前更适合先做基础治理梳理”。 |
| P2 | 摘要需要避免过长 | 明确摘要不包含 24 道题逐题答案，只包含总分、8 维度、Top 3 和建议。 |

### 5.7 创始人本人

结论：通过。

认可点：

1. V1.5 能把润序科技的 PMO 和项目治理能力产品化表达。
2. 复制摘要能让首次沟通更聚焦，减少重复访谈成本。
3. 不做后台符合当前阶段，先验证客户是否愿意带着结果咨询。

问题与建议：

| 优先级 | 问题 | 修改要求 |
|---|---|---|
| P1 | 用户可能以为问卷建议已经是完整解决方案 | 在页面和摘要中增加“该结果用于诊断沟通，不是完整解决方案”。 |
| P2 | 咨询价值承接不足 | 在联系区增加：“请将诊断摘要通过飞书或邮箱发送给我们，便于快速判断项目治理重点。” |

### 5.8 技术负责人 / 架构

结论：有条件通过。

认可点：

1. V1.5 架构边界简单，不引入后端、数据库和外部通知。
2. 评分模型可封装为稳定数据结构，为 V2 结果对象预留空间。
3. 不引入第三方图表库，有利于安全和性能控制。

问题与建议：

| 优先级 | 问题 | 修改要求 |
|---|---|---|
| P1 | 无外部请求边界需要更硬 | PRD 明确 V1.5 不新增接口、不接入统计脚本、不发送网络请求。 |
| P1 | 数据结构需统一 | 明确页面显示、雷达图、Top 3、摘要必须共用同一计算结果。 |
| P2 | 性能要求不够量化 | 增加要求：24 个问题变更后计算和图表更新应无明显卡顿；不引入超过现有静态资源规模的大型依赖。 |

## 8. 可直接写入 PRD 的建议

### 8.1 CTA 交互补充

```text
用户点击“复制诊断摘要”后，系统将当前总分、参考 CMMI 映射、8 维度得分、Top 3 短板和治理建议写入剪贴板。
复制成功后按钮文案变为“已复制”，保持 1.5-2 秒后恢复为“复制诊断摘要”。
用户点击“生成诊断摘要并预约”后，跳转至 `index.html#contact`。
页面提示用户先复制诊断摘要，再通过飞书或邮箱发送给润序科技。
```

### 8.2 雷达图 0 分状态补充

```text
页面初始状态下，8 个维度均为 0 分。
雷达图仍需展示坐标网格、8 个维度标签和中心点，并显示说明文案：“当前各维度均为 0 分”。
用户调整任一题目后，雷达图基于最新维度分实时重绘。
Canvas 实现需适配 `devicePixelRatio`，并在窗口尺寸变化时重绘。
```

### 8.3 数据与隐私补充

```text
V1.5 所有评分和摘要生成均在浏览器本地完成。
本版本不新增后端接口，不保存问卷答案，不上传用户数据，不接入统计埋点，不向飞书、邮箱或第三方服务自动发送结果。
诊断摘要仅在用户主动点击复制后写入本地剪贴板。
```

## 9. 最终评审结论

V1.5 PRD 当前可进入研发评审和开发排期，但建议先按 P1 清单做一次 PRD 修订。

如果不修订直接开发，主要风险不是功能做不出来，而是实现口径不一致：雷达图初始状态、复制反馈、CMMI 合规表达、无后台边界和 CTA 跳转容易出现偏差。
