# 北京润序科技官网 V1.5 诊断结果 PRD 多角色评审记录

需求类型：存量功能增强 / 售前诊断转化增强 / 前端本地计算工具  
评审角色：

## 1. 总体结论

结论：有条件通过，可以进入 V1.5 研发。

原因：

1. 需求目标清楚：从“总分展示”升级为“可诊断、可复制、可预约”的前端闭环。
2. 范围控制合理：V1.5 不建设后端、不保存数据，符合当前静态官网技术方案。
3. 关键功能完整：雷达图、8 维度得分、短板 Top 3、治理建议、复制摘要均已定义。
4. 项目经理承接链路更清晰：用户可把结构化摘要发给项目经理，避免只传一个总分。

进入研发前建议补齐 3 个 P1：

1. 明确雷达图移动端展示尺寸和降级方式。
2. 明确“复制失败”的手动摘要区域样式与位置。
3. 明确摘要是否包含每题答案；V1.5 建议不包含每题答案，只包含维度得分和短板建议，避免摘要过长。

## 2. 需求可行性评估

| 维度 | 结论 | 说明 |
|---|---|---|
| 功能完整性 | ✅ 可行 | 覆盖总分、维度得分、雷达图、短板建议、摘要复制、预约跳转 |
| 可测试性 | ✅ 可行 | 0 分、100 分、单维度高分、同分排序、复制成功/失败均可测试 |

综合结论：可进入研发，需补充复制失败降级细节和移动端展示规则。

## 3. 模块拆解清单

1. 雷达图模块（P0）
   - 功能：展示 8 个治理维度的 0-100 分雷达图。
   - 数据流：读取 24 个 select 值 → 聚合 8 个维度分 → 绘制 Canvas/SVG。

2. 维度得分模块（P0）
   - 功能：展示每个维度名称、得分、状态。
   - 数据流：维度分数 → 状态规则 → 列表渲染。

3. 短板 Top 3 模块（P0）
   - 功能：按维度得分从低到高识别短板 Top 3。
   - 数据流：维度分数排序 → 取前三 → 输出建议。

4. 治理建议模块（P0）
   - 功能：每个短板维度展示对应治理建议。
   - 数据流：维度 key → 建议字典。

5. 诊断摘要模块（P0）
   - 功能：生成结构化文本，包含总分、等级、8 维度、短板和免责声明。
   - 数据流：总分 + 维度分 + Top 3 → 文本模板。

6. 复制摘要模块（P1）
   - 功能：复制诊断摘要到剪贴板；失败时展示手动复制文本。
   - 数据流：摘要文本 → Clipboard API → 成功/失败状态。

7. CTA 转化模块（P1）
   - 功能：将 CTA 改为“生成诊断摘要并预约”，跳转联系区。
   - 数据流：点击 → 跳转 `index.html#contact`。

## 4. 产品负责人评审

评审结论：通过，V1.5 的产品价值明确。

认可点：

- 解决了“只有总分无法诊断”的核心问题。
- 通过雷达图和短板 Top 3，把评测结果转成项目经理可用信息。
- 不做后端，避免 V1.5 范围膨胀。
- 复制摘要是当前阶段最小可行闭环。

问题：

|---|---|---|
| P1 | 摘要长度可能过长 | V1.5 摘要只放总分、8 维度、Top 3 和建议，不放 24 题明细 |
| P1 | CTA 动作顺序需更明确 | 页面提示“先复制诊断摘要，再预约诊断” |
| P2 | 没有后台仍无法自动统计转化 | V1.5 可接受，V2 再做结果保存和通知 |

产品结论：可以进入研发。

## 5. UI/UX 设计评审

评审结论：有条件通过。

认可点：

- 雷达图适合展示 8 维度强弱。
- 维度列表可以弥补雷达图数值不精确的问题。
- 短板 Top 3 对用户决策和项目经理诊断都有帮助。

问题：

| 优先级 | 问题 | 建议 |
|---|---|---|
| P1 | 左侧评分面板内容会变多，可能过长 | 雷达图、维度列表、短板建议需要分层；移动端改为纵向卡片 |
| P1 | 雷达图在 375px 可能看不清标签 | 移动端雷达图缩小，主要依赖维度列表展示数值 |
| P1 | 复制失败时的摘要区没有视觉方案 | 增加一个折叠/展开的文本区域，用于手动复制 |
| P2 | “生成诊断摘要并预约”按钮较长 | 移动端可显示“复制摘要并预约” |

设计结论：研发前补移动端展示细节。

## 6. 前端研发评审

评审结论：通过。

认可点：

- 原生 Canvas/SVG 可以实现雷达图，不需要图表库。
- 当前问卷已有 24 个 select，维度聚合实现成本低。
- 文本摘要生成和剪贴板复制都是前端可完成。
- 不涉及后端，风险可控。

问题：

| 优先级 | 问题 | 建议 |
|---|---|---|
| P1 | Clipboard API 可能被浏览器策略拦截 | 使用 `navigator.clipboard.writeText`，失败时展示 textarea |
| P1 | Canvas 雷达图需考虑高清屏 | 按 `devicePixelRatio` 缩放绘制 |
| P1 | 维度顺序和 select name 绑定需稳定 | 明确 A-H 维度映射，避免依赖 DOM 文案 |
| P2 | 当前问卷 JS 逻辑会变复杂 | 可拆出维度配置、建议配置和渲染函数 |

前端结论：可以开发。

## 7. 后端研发评审

评审结论：V1.5 不需要后端，边界明确。

认可点：

- PRD 明确不保存、不上传、不生成用户 ID。
- 复制摘要由用户主动触发，不涉及服务端数据处理。
- V2 后台承接已有方向，包括结果 ID、保存接口、项目经理查看入口、飞书通知。

问题：

| 优先级 | 问题 | 建议 |
|---|---|---|
| P1 | 如果后续要自动提交摘要，必须重新评审 | V1.5 禁止隐式上传 |
| P1 | 如果接入飞书 Webhook，不能放前端 | V2 必须走服务端中转 |
| P2 | V2 数据结构可提前设计 | 但不进入 V1.5 范围 |

后端结论：V1.5 不建设后端，范围合理。

## 13. 必须修改的问题

### P1

1. [复制摘要] 复制失败后的手动复制区域未定义具体样式和位置  
   → 影响：前端实现和测试无法判断失败态是否合格  
   → 修改要求：补充 textarea 或弹层展示规则。

2. [移动端] 雷达图在小屏下的展示策略不够明确  
   → 影响：375px 可能出现可读性问题  
   → 修改要求：明确移动端雷达图尺寸、是否隐藏标签、是否依赖维度列表。

3. [摘要内容] 是否包含 24 题明细未明确排除  
   → 影响：摘要可能过长  
   → 修改要求：V1.5 明确不包含 24 题明细。

### P2

1. [文案] 复制后提示文案可再打磨。
2. [V2] 后台结果数据结构可提前设计，但不进入本期。

V1.5 需求可以进入研发。

3. 摘要不包含 24 题明细的限制。
