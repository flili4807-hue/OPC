# 北京润序科技官网 V1.5 诊断结果增强设计方案

版本：V1.5 修订版  
日期：2026-07-14  
设计角色：UI/UX 设计师 / B2B 咨询转化设计师  
对应 PRD：`北京润序科技官网V1.5诊断结果PRD.md`  
关联页面：`OPC/网站/runxu-tech/questionnaire.html`
修订依据：`北京润序科技官网V1.5诊断结果设计方案_多角色评审记录.md`

## 1. 设计结论

V1.5 不重新设计整站，也不改变当前深海蓝、高端科技感的品牌方向。

本版本只聚焦问卷页的结果表达能力：把“一个总分”升级为“总分 + 雷达图 + 8 维度得分 + Top 3 短板 + 可复制诊断摘要”。

核心设计判断：

1. 用户需要先看懂自己哪里弱，再愿意预约诊断。
2. 项目经理需要拿到维度数据和治理建议，而不是只拿到一个分数。
3. 当前没有后台，所以页面必须提供“复制诊断摘要”作为数据传递方案。
4. 雷达图承担快速感知，维度列表承担准确阅读，摘要承担咨询交接。

本次修订吸收多角色评审意见，重点调整：

1. 降低左侧结果栏信息密度，避免桌面端首屏拥挤。
2. 明确“先复制诊断摘要，再预约诊断”的转化引导。
3. 补齐雷达图 0 分状态，避免用户误以为图表异常。
4. 明确 Canvas 响应式、高清屏和窗口变化处理。
5. 严格保持 V1.5 不上传、不保存、不埋点、不外发数据。

## 2. 设计目标

### 2.1 用户侧目标

用户完成问卷后，应在 10 秒内理解：

- 我的整体项目成功率得分是多少。
- 当前风险等级和 CMMI 映射是什么。
- 8 个治理维度中哪些明显偏弱。
- 最优先治理的 3 个问题是什么。
- 下一步如何把结果发给润序科技预约诊断。

### 2.2 项目经理侧目标

项目经理收到用户复制的摘要后，应能快速判断：

- 当前项目风险集中在哪些维度。
- 首次诊断会优先问哪些问题。
- 是否需要先做目标范围、组织职责、需求变更、计划风险、质量门禁、数据效能、工具流程或复盘机制治理。
- 该客户是否适合进入后续项目诊断或 PMO 体系搭建服务。

### 2.3 转化侧目标

V1.5 的转化不是“直接提交表单”，而是：

```text
完成评测
→ 看懂结果
→ 复制诊断摘要
→ 带摘要联系润序科技
→ 预约 30 分钟项目诊断
```

因此按钮和页面结构必须引导用户先复制摘要，再预约诊断。

## 3. 页面整体结构

问卷页保持左右两栏结构，但左侧结果卡升级为“诊断结果中心”。修订后桌面端左侧只承载最关键的诊断信息，完整 8 维度列表放到右侧问卷下方或结果明细区。

```text
顶部导航
└─ 项目成功率自评问卷说明

主体区域
├─ 左侧固定结果栏
│  ├─ 总分与风险等级
│  ├─ 雷达图
│  ├─ Top 3 短板
│  ├─ 复制诊断摘要
│  └─ 生成诊断摘要并预约
│
└─ 右侧问卷区
   ├─ A. 目标与范围
   ├─ B. 组织与职责
   ├─ C. 需求与变更
   ├─ D. 计划与风险
   ├─ E. 质量与测试
   ├─ F. 数据与效能
   ├─ G. 工具链与流程
   └─ H. 复盘与持续改进

结果明细区
├─ 8 维度得分列表
└─ 诊断摘要手动复制区
```

## 5. 核心模块设计

### 5.1 总分结果卡

位置：左侧结果栏顶部。

内容：

- 标题：`项目成功率得分`
- 大数字：`0-100`
- 风险等级：如 `极高风险 · CMMI L1`
- 总体建议：一句话说明当前优先治理方向。
- 成熟度区间：85+、70-84、55-69、40-54、0-39，可作为轻量标签展示，不占用过大空间。

设计要求：

1. 总分使用荧光绿，保证一眼可见。
2. 风险等级紧贴总分，不要放到远处。
3. 风险等级展示建议使用 `参考映射：CMMI L1`，避免被理解为官方评级。
4. 初始 0 分必须看起来是一个有效诊断状态，而不是未加载状态。
5. 总体建议控制在 2 行以内，避免压缩雷达图空间。
6. 风险文案要保持建设性，例如：`当前更适合先做基础治理梳理。`

### 5.2 雷达图模块

位置：总分结果卡下方。

目的：让用户快速看到 8 个维度的相对强弱。

展示规则：

- 8 个轴：目标、组织、需求、计划、质量、数据、工具、复盘。
- 初始全部为 0，图形中心收缩，但仍显示坐标网格、8 个维度标签和中心点。
- 用户选择分值后实时更新。
- 雷达填充使用半透明蓝色，边缘使用电光蓝。
- 坐标轴和网格线使用低透明度浅蓝。

设计要求：

1. 雷达图不要过大，避免抢走问卷操作区域。
2. 移动端雷达图置于总分下方，并缩小到 280-320px。
3. 维度标签使用短词，不使用完整长标题，避免拥挤。
4. 雷达图只负责趋势判断，具体分数交给维度列表。
5. 0 分状态下增加说明：`当前各维度均为 0 分`，避免用户误以为图表未加载。
6. Canvas 使用固定 CSS 尺寸和 `devicePixelRatio` 放大绘制，窗口变化时重绘。

### 5.3 8 维度得分列表

位置：桌面端放在右侧问卷区下方的结果明细区；移动端放在问卷下方展开。

不再默认放入左侧结果栏，避免左侧信息密度过高。

每个维度展示：

- 维度名称
- 得分
- 状态标签
- 短进度条

状态规则：

| 分数 | 状态 | 色彩建议 |
|---:|---|---|
| 85-100 | 优势维度 | 荧光绿 |
| 70-84 | 稳定维度 | 蓝色 |
| 55-69 | 待加强 | 浅蓝/黄绿 |
| 40-54 | 风险维度 | 橙色 |
| 0-39 | 核心短板 | 红橙 |

设计要求：

1. 8 个维度必须可完整阅读，不应只依赖雷达图。
2. 分数和状态要靠近维度名称，避免用户来回找。
3. 进度条只做辅助，不能替代分数。
4. 状态颜色落为明确 class，建议：`is-strong`、`is-stable`、`is-warning`、`is-risk`、`is-critical`。

### 5.4 Top 3 短板模块

位置：左侧结果栏中下部，紧跟雷达图。

内容：

- 模块标题：`优先治理 Top 3`
- 每条包含：
  - 排名
  - 维度名称
  - 得分
  - 一条治理建议

初始状态：

```text
1. 目标与范围 0：优先明确业务目标、用户目标、成功指标、本期范围和不做事项。
2. 组织与职责 0：建立项目 Owner、RACI 职责矩阵、跨团队依赖响应机制。
3. 需求与变更 0：建立 PRD/原型/技术方案准入，明确变更评估、审批和排期影响。
```

设计要求：

1. Top 3 是项目经理最重要的信息，优先级高于完整 8 维列表。
2. 每条建议控制在 1-2 行。
3. 排名使用小圆点或数字徽标，强化优先级。
4. 不使用大段解释，避免变成报告页。
5. 同分时按 A-H 顺序排序，保证展示稳定。

### 5.5 复制诊断摘要模块

位置：左侧结果栏底部，主 CTA 上方。

按钮：

- 次级按钮：`复制诊断摘要`
- 主按钮：`生成诊断摘要并预约`
- 辅助说明：`建议先复制诊断摘要，再通过飞书或邮箱发送给我们。`

交互：

1. 点击 `复制诊断摘要` 后，按钮临时变为 `已复制`。
2. `已复制` 状态保持 1.5-2 秒后恢复为 `复制诊断摘要`。
3. 如果复制失败，在按钮下方展示一个可手动复制的文本框。
4. 点击 `生成诊断摘要并预约` 后跳转首页联系区：`index.html#contact`。

设计要求：

1. 两个按钮不能都做成强主按钮。
2. `复制诊断摘要` 是必要动作，采用描边或半透明按钮。
3. `生成诊断摘要并预约` 是转化动作，采用蓝色渐变主按钮。
4. 按钮旁边或下方保留一句说明：`V1.5 不上传数据，请复制摘要后通过飞书或邮箱发送。`
5. 移动端按钮可缩短为 `复制摘要` 和 `预约诊断`，但语义必须保持一致。

### 5.6 手动复制文本区

触发条件：浏览器阻止剪贴板写入。

展示内容：

- 标题：`请手动复制以下诊断摘要`
- 文本框：自动填充完整摘要
- 辅助说明：`当前浏览器未开放剪贴板权限，内容未上传。`

设计要求：

1. 默认隐藏，不占用页面空间。
2. 展开后要有足够高度，至少显示 6-8 行文本。
3. 文本框使用等宽字体或紧凑无衬线，方便复制。
4. DOM 位置固定在复制按钮下方，避免展开后挤压雷达图和 Top 3。

## 6. 交互状态设计

### 6.1 初始状态

页面打开后：

- 所有题目默认 `0 分`。
- 总分为 `0`。
- 等级为 `极高风险 · 参考映射：CMMI L1`。
- 雷达图所有维度为 0。
- 雷达图显示网格、中心点和说明 `当前各维度均为 0 分`。
- Top 3 为 A/B/C。
- 复制摘要按钮可点击。

设计解释：

0 分不是空状态，而是“尚未建立治理能力”的有效状态。默认 0 分也允许复制诊断摘要，摘要应体现当前为 `极高风险 · 参考映射：CMMI L1`。

### 6.2 实时更新状态

用户调整任一题目后：

- 总分实时变化。
- 对应维度分实时变化。
- 雷达图实时变化。
- Top 3 重新排序。
- 诊断摘要同步更新。

动效建议：

- 分数变化可使用轻微数字过渡。
- 雷达图变化使用即时重绘，不需要复杂动画。
- Top 3 变化不做明显跳动，避免干扰填写。

### 6.3 全部完成状态

当用户主动调整题目后：

- 不强制显示 `已完成 24/24`，因为默认 0 分无法判断用户是否真实填写过。
- 如需显示进度，应基于“用户已主动修改过的题目数量”，而不是基于是否有默认值。
- CTA 文案保持 `生成诊断摘要并预约`。
- 不额外弹窗，避免打断。

### 6.4 移动端状态

移动端布局：

```text
顶部导航
问卷说明
总分结果卡
雷达图
Top 3 短板
复制摘要 / 预约按钮
问卷题目
8 维度得分列表
手动复制文本区
```

移动端要求：

1. 禁止横向滚动。
2. 下拉框宽度不超过屏幕。
3. 雷达图控制在屏幕宽度内。
4. 左侧固定栏在移动端取消固定，改为普通纵向模块。
5. 主按钮文案过长时可缩短为 `预约项目诊断`，按钮附近保留“先复制摘要”的说明。

高分状态：


| 成功反馈 | 已复制 |

| 移动端主 CTA | 预约项目诊断 |

## 8. 诊断摘要设计

摘要格式应保持短、清晰、可转发。

```text
【Runxu Tech 项目成功率自评摘要】
总分：42
等级：高风险 · 参考映射：CMMI L1-L2

8 维度得分：
- 目标与范围：40
- 组织与职责：53
- 需求与变更：33
- 计划与风险：47
- 质量与测试：47
- 数据与效能：38
- 工具链与流程：52
- 复盘与持续改进：44

短板 Top 3：
1. 需求与变更 33：建立 PRD/原型/技术方案准入，明确变更评估、审批和排期影响。
2. 数据与效能 38：建立进度、质量、效率、风险指标和可视化效能看板。
3. 目标与范围 40：优先明确业务目标、用户目标、成功指标、本期范围和不做事项。

说明：本工具用于项目管理诊断沟通，不等同于官方 CMMI 评估或认证。
```

设计要求：

1. 摘要不包含 24 道题的逐题答案，避免太长。
2. 摘要必须包含 8 维度得分，方便项目经理判断。
3. 摘要必须包含免责声明。
4. 摘要文案不使用营销语，保持诊断口吻。
5. 摘要不包含 24 道题的逐题答案，避免内容过长。

## 9. 可访问性与可读性

必须满足：

1. 深色背景下正文对比度足够，避免浅蓝文字过淡。
2. 下拉框默认 `0 分` 清晰可见。
3. 雷达图不能作为唯一信息来源，必须有维度列表。
4. CTA 文案明确，不让用户误以为系统已自动提交数据。
5. 移动端 375px 下不出现横向滚动。
6. 雷达图不能作为唯一判断依据，8 维度列表必须可读。

## 10. 开发交付说明

前端实现建议：

1. 雷达图使用 Canvas 原生绘制，不引入图表库。
2. 8 维度数据结构与问卷题目编号绑定，便于后续 V2 接后台。
3. 页面本地计算，不发起任何网络请求。
4. 复制摘要使用浏览器 Clipboard API。
5. Clipboard API 失败时展示手动复制文本框。
6. 页面显示、Top 3、雷达图和摘要生成必须共用同一套计算结果。
7. Canvas 在页面初始化、分值变化和窗口 resize 时重绘。
8. V1.5 不接入统计脚本、不新增埋点、不新增任何外部提交。

数据结构建议：

```js
dimensions = [
  {
    key: 'A',
    name: '目标与范围',
    short: '目标',
    fields: ['A1', 'A2', 'A3'],
    advice: '优先明确业务目标、用户目标、成功指标、本期范围和不做事项。',
    score: 0,
    status: '核心短板'
  }
```

## 11. 设计验收标准

### 11.1 视觉验收

- [ ] 问卷页整体仍保持深海蓝科技风格。
- [ ] 不出现白底低对比问题。
- [ ] 总分、风险等级、雷达图、Top 3 的视觉主次清晰。
- [ ] 按钮区能明确区分“复制摘要”和“预约诊断”。
- [ ] 左侧结果栏不塞入完整 8 维列表，首屏不拥挤。
- [ ] 雷达图 0 分状态显示网格、中心点和说明。

### 11.2 交互验收

- [ ] 页面初始状态 0 分是明确可读的诊断状态。
- [ ] 修改任一题目后，总分、雷达图、维度列表、Top 3 同步更新。
- [ ] 点击复制后有成功反馈。
- [ ] 复制成功反馈 1.5-2 秒后恢复。
- [ ] 复制失败时出现手动复制区域。
- [ ] 主 CTA 跳转到 `index.html#contact`。
- [ ] 同分排序时 Top 3 按 A-H 顺序展示。
- [ ] 375px、768px、1440px 三档布局均不溢出。

### 11.3 内容验收

- [ ] 8 个维度名称与 PRD 一致。
- [ ] Top 3 建议与 PRD 治理建议一致。
- [ ] 诊断摘要包含总分、等级、8 维度得分、短板 Top 3、建议和免责声明。
- [ ] 页面明确说明 V1.5 不上传、不保存数据。
- [ ] 风险文案保持建设性，不只输出负面结论。
- [ ] CMMI 表达使用“参考映射”，避免被误解为正式认证。
