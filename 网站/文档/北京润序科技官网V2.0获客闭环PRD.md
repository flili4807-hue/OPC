# 北京润序科技官网 V2.0 获客闭环 PRD

版本：V2.0 修订版  
日期：2026-07-14  
产品负责人：北京润序科技官网产品经理 Agent  
需求类型：官网获客转化 / 留资表单 / 诊断报告 / 邮件与飞书通知 / 后端线索承接  
关联页面：`questionnaire.html`、`index.html#contact`  
上游版本：`北京润序科技官网V1.5诊断结果PRD.md`
修订依据：`北京润序科技官网V2.0获客闭环PRD_多角色评审记录.md`
P1 修订依据：`北京润序科技官网V2.0获客闭环PRD与设计方案_联合评审记录.md`

## 1. 背景

V1.5 已经完成项目成功率自评问卷的前端诊断闭环：

1. 24 个问题默认 0 分。
2. 8 维度得分。
3. 雷达图。
4. Top 3 短板。
5. 诊断摘要复制。
6. 跳转联系区。

但 V1.5 的获客动线仍存在业务缺口：

1. 用户可以查看诊断结果，但官网无法获取客户信息。
2. 用户需要主动复制摘要并通过飞书或邮箱联系，流失率高。
3. 咨询线索无法沉淀，无法后续跟进、复盘和评估转化效果。
4. 创始人 / 售前无法自动收到用户的诊断结果和联系方式。
5. 无法判断哪些用户完成了评测、哪些用户有真实咨询意向。

因此 V2.0 需要从“本地诊断工具”升级为“获客闭环工具”。

## 2. 产品目标

### 2.1 业务目标

1. 将完成问卷的高意向用户转化为可跟进线索。
2. 在用户查看项目管理自评诊断报告前完成必要留资。
3. 自动把用户联系方式、项目背景和诊断结果发送给润序科技。
4. 降低人工等待用户主动联系的流失。
5. 为后续售前诊断、客户分层、转化率分析和线索管理打基础。

### 2.2 用户目标

用户完成问卷后，应能够：

1. 明确知道自己即将生成一份项目管理自评诊断报告。
2. 用较少字段提交咨询信息。
3. 在授权后查看项目管理自评诊断报告。
4. 知道润序科技会基于报告联系自己。
5. 有清晰的隐私和授权预期。

### 2.3 售前 / 创始人目标

售前或创始人收到线索后，应能快速知道：

1. 咨询者是谁。
2. 来自哪个公司或团队。
3. 联系方式是什么。
4. 项目背景是什么。
5. 项目成功率总分、风险等级和 CMMI 参考映射。
6. 8 个治理维度分别得分。
7. Top 3 短板和建议。
8. 是否值得优先跟进。

## 3. 本期范围

### 3.1 V2.0-MVP 第一阶段包含

第一阶段以“可稳定留资、可邮件触达、可解锁报告”为准，不把范围扩散到完整 CRM 或后台系统。

9. 隐私与合规提示。

## 4. 核心用户路径

```text
用户进入问卷页
→ 完成或调整 24 道题
→ 页面实时展示总分、雷达图、8 维得分、Top 3
→ 用户点击“生成项目管理自评诊断报告”
→ 页面弹出咨询者信息表单
→ 用户填写信息并勾选授权
→ 用户点击“提交并生成报告”
→ 前端调用线索提交接口
→ 后端保存线索和诊断结果
→ 后端发送邮件到 Laura_fan@126.com
→ 若已启用飞书通知，后端发送飞书通知
→ 前端展示项目管理自评诊断报告
→ 用户可复制报告摘要或返回首页
```

## 5. 页面与交互需求

### 5.1 CTA 调整

V1.5 当前 CTA：

- `复制诊断摘要`
- `生成诊断摘要并预约`

V2.0 调整为：

- 主 CTA：`生成项目管理自评诊断报告`
- 次级动作：`复制当前摘要`

规则：

1. 用户点击 `生成项目管理自评诊断报告` 后，不直接展示完整报告。
2. 先展示留资表单。
3. 表单提交成功后，再展示项目管理自评诊断报告。
4. 如果用户不愿留资，仍可复制当前摘要，但不展示完整报告。

### 5.2 留资表单

#### 展示时机与形态

用户点击 `生成项目管理自评诊断报告` 后展示留资表单。

展示形态：

1. 桌面端：居中弹窗。
2. 移动端：整页卡片，优先保证字段可读、键盘输入和提交按钮可达。
3. 弹窗关闭后，用户仍停留在当前问卷结果页，可继续复制当前摘要。
4. 本期采用页面会话解锁：提交成功后在当前页面展示报告；刷新页面后回到未解锁状态，不做长期解锁凭证。

#### 字段

| 字段 | 类型 | 是否必填 | 规则 | 用途 |
|---|---|---|---|---|
| 姓名 | 文本 | 必填 | 2-20 字 | 识别咨询者 |
| 公司 / 团队 | 文本 | 必填 | 2-50 字 | 判断客户背景 |
| 职位 / 角色 | 文本 | 选填 | 0-30 字 | 判断沟通对象角色 |
| 联系方式 | 文本 | 必填 | 手机 / 邮箱 / 飞书号至少一种有效格式 | 后续联系 |
| 项目背景 | 多行文本 | 必填 | 10-500 字 | 判断诊断价值 |
| 授权勾选 | checkbox | 必填 | 必须勾选 | 合规授权 |

项目背景输入提示：

```text
请简单描述项目类型、当前阶段、主要问题或你希望优先诊断的方向。
```

#### 授权文案

```text
我同意北京润序科技有限责任公司基于本次评测结果和联系方式，用于项目诊断沟通。该结果不等同于官方 CMMI 评估或认证。
```

### 5.3 表单校验

| 场景 | 规则 |
|---|---|
| 姓名为空 | 提示：请填写姓名 |
| 公司 / 团队为空 | 提示：请填写公司或团队名称 |
| 联系方式为空 | 提示：请填写手机号、邮箱或飞书号 |
| 联系方式格式明显无效 | 提示：请填写有效联系方式 |
| 项目背景少于 10 字 | 提示：请简单说明项目背景或当前困扰 |
| 未勾选授权 | 提示：请确认授权后生成诊断报告 |
| 重复点击提交 | 按钮进入提交中，禁止重复提交 |

联系方式校验：

1. 邮箱：包含 `@` 和有效域名结构。
2. 手机：支持中国大陆 11 位手机号。
3. 飞书号：允许 4-50 位字母、数字、下划线、短横线或用户填写“飞书同手机号 / 飞书同邮箱”。
4. 只要求联系方式字段至少满足一种可联系格式，不强制拆成多个字段。

### 5.4 项目管理自评诊断报告展示

后端保存线索成功后展示完整报告。V2.0 的报告复用 V1.5 结果区，并新增“已提交咨询信息”和“下一步诊断建议”。

解锁条件：

1. 服务端校验通过。
2. 线索和诊断结果保存成功。
3. 邮件或飞书通知失败不阻断报告解锁，但后端必须记录 `notifyStatus`。
4. 线索保存失败时不解锁报告，提示用户重试或复制摘要联系。
5. 本期只做页面会话解锁，刷新后回到未解锁状态。

#### 解锁前展示

用户未提交留资前可查看：

1. 总分。
2. 风险等级。
3. CMMI 参考映射。
4. 雷达图。
5. Top 3 短板标题和维度名称。
6. 简短提示：`提交咨询信息后查看项目管理自评诊断报告和治理建议。`

用户未提交留资前不展示：

1. Top 3 完整治理建议。
2. 完整 8 维度解释。
3. 完整报告摘要。

#### 解锁后展示

报告内容：

1. 总分。
2. 风险等级。
3. CMMI 参考映射。
4. 8 维度雷达图。
5. 8 维度分数列表。
6. Top 3 短板。
7. Top 3 治理建议。
8. 下一步建议：预约 30 分钟项目诊断。
9. 免责声明：本工具用于项目管理诊断沟通，不等同于官方 CMMI 评估或认证。
10. 已提交提示：`我们已收到你的咨询信息，将基于评测结果与你联系。`
11. 建议跟进动作：基于 Top 3 短板生成一句售前建议。

报告不展示：

1. 后端保存 ID。
2. 内部线索评分。
3. 邮件或飞书发送状态详情。

### 5.5 提交成功状态

成功后页面提示：

```text
项目管理自评诊断报告已生成，我们已收到你的咨询信息。润序科技将基于评测结果与你联系，通常会在 1-2 个工作日内响应。
```

按钮：

- `复制报告摘要`
- `返回首页`

### 5.6 提交失败状态

失败原因包括：

1. 网络异常。
2. 后端接口异常。
3. 邮件发送失败。
4. 飞书通知失败。
5. 服务端校验失败。

前端展示：

```text
提交失败，请稍后重试。你也可以复制诊断摘要，通过飞书或邮箱联系我们。
```

按钮：

- `重试提交`
- `复制诊断摘要`
- `通过邮箱联系`

```json
  "consent": true,
  "score": 42,
  "level": "高风险",
  "cmmiReference": "参考映射：CMMI L1-L2",
  "dimensions": [
    {
      "key": "A",
      "name": "目标与范围",
      "score": 40,
      "status": "风险维度"
    }

    }
  ],

    "A1": 2,
    "A2": 1
  },
  "notifyStatus": {
    "email": "success",
    "feishu": "not_configured"
  },
  "duplicate": false,
  "leadQuality": "high"
}
```

### 6.2 数据保存要求

1. 必须保存用户授权状态。
2. 必须保存提交时间。
3. 必须保存完整问卷答案，便于售前复盘。
4. 必须保存总分、8 维得分和 Top 3。
5. 不保存用户 IP 的完整明文；如需风控，只保存脱敏或哈希值。
6. 数据保存周期建议 180 天，后续可配置。
7. 同一联系方式 10 分钟内重复提交时仍保存记录，但标记 `duplicate=true`，不直接丢弃。
8. 线索质量字段 `leadQuality` 取值为 `high`、`medium`、`unknown`。
9. MVP 必须保存到后端可查询的数据介质中，不允许只依赖邮件通知。
10. 具体保存介质由技术方案确认，优先建议 SQLite 或 PostgreSQL；临时 JSONL 仅允许用于本地开发或演示环境。

### 6.3 线索质量分层

| 线索等级 | 判断规则 | 售前建议 |
|---|---|---|
| high | 项目背景不少于 30 字，且总分低于 70 或 Top 3 存在 0 分维度 | 24 小时内优先跟进 |
| medium | 项目背景不少于 10 字，联系方式有效 | 1-2 个工作日内跟进 |
| unknown | 信息完整但项目背景较弱或重复提交 | 视情况跟进 |

线索等级只用于内部判断，不展示给用户。

## 7. 后端接口需求

### 7.1 提交线索接口

接口：

```text
POST /api/leads/diagnostic
```

部署环境：

| 环境 | Base URL |
|---|---|
| 本地开发 | `http://127.0.0.1:8765/api` 或本地后端代理 |
| 测试环境 | 待技术方案确定 |
| 生产环境 | 与官网同域名 `/api` 优先 |

最终 Base URL 由 V2.0 技术方案确认。

请求体：

```json
{
  "profile": {
    "name": "张三",
    "company": "某科技公司",
    "role": "产品负责人",
    "contact": "zhangsan@example.com",
    "projectBackground": "当前项目跨团队协作复杂，需求变更频繁，版本延期明显。",
    "consent": true
  },
  "diagnostic": {
    "score": 42,
    "level": "高风险",
    "cmmiReference": "参考映射：CMMI L1-L2",
    "dimensions": [],
    "weaknesses": [],
    "answers": {}
  },
  "clientMeta": {
    "page": "questionnaire.html",
    "userAgent": "由浏览器提供",
    "submittedAt": "2026-07-14T16:24:00+08:00"
  }
}
```

响应成功：

```json
{
  "success": true,
  "leadId": "lead_20260714_xxxxxx",
  "reportUnlocked": true,
  "duplicate": false,
  "notifyStatus": {
    "email": "success",
    "feishu": "not_configured"
  }
}
```

响应失败：

```json
{
  "success": false,
  "code": "VALIDATION_ERROR",
  "message": "请填写有效联系方式"
}
```

### 7.2 接口规则

1. 只接受 `POST`。
2. 必须校验授权 `consent=true`。
3. 必须做服务端字段校验，不能只依赖前端。
4. 必须做基础频率限制。
5. 同一联系方式 10 分钟内重复提交，可允许但需要标记为重复线索。
6. 接口不能向前端返回内部邮件或飞书错误详情。
7. 重复提交规则：同一联系方式 10 分钟内重复提交时，保存记录并返回 `duplicate=true`，邮件标题和飞书卡片标记“重复线索”。
8. 前端提交中按钮必须禁用，后端仍需做重复标记，不能只依赖前端防重复。
9. 接口成功标准为“服务端校验通过 + 线索保存成功”。
10. 邮件或飞书通知失败不阻断 `reportUnlocked=true`，但必须写入 `notifyStatus`。
11. 线索保存失败时返回失败，前端不得解锁报告。

### 8.4 邮件服务方案

4. 邮件发送失败不直接向前端暴露内部错误。
5. 如果线索保存成功但邮件发送失败，前端仍解锁报告，后端记录 `notifyStatus.email=failed`。

## 9. 飞书通知需求

### 9.1 触发方式

后端接口保存线索后，调用飞书机器人 Webhook 或飞书开放平台能力发送通知。

V2.0-MVP 中飞书通知为可配置能力：

1. 如果飞书机器人 Webhook 已配置，则提交后同步或异步发送飞书通知。
2. 如果未配置，`notifyStatus.feishu=not_configured`，不阻断用户查看报告。
3. 飞书 Webhook 只能存放在后端环境变量，不能暴露到前端。

### 9.2 飞书消息内容

建议使用卡片消息。

卡片内容：

```text
新官网项目诊断线索

线索摘要：高风险 / high / 非重复
建议跟进动作：优先确认需求变更机制和发布门禁现状。
建议首问问题：当前项目延期或返工最集中发生在哪个阶段？

姓名：张三
公司：某科技公司
角色：产品负责人
联系方式：zhangsan@example.com
项目背景：当前项目跨团队协作复杂...

总分：42
等级：高风险
CMMI：参考映射 CMMI L1-L2

Top 3 短板：
1. 需求与变更 33

```

### 9.3 失败处理

1. 飞书通知失败不应阻断用户生成报告。
2. 后端需记录通知失败状态。
3. 邮件发送成功但飞书失败时，前端仍展示提交成功。
4. 后续可通过日志补偿处理。

## 10. 权限、隐私与合规

### 10.1 用户授权

用户必须勾选授权后才能提交。

授权文案必须清晰说明：

1. 收集联系方式。
2. 收集评测结果。
3. 用途是项目诊断沟通。
4. 不等同于官方 CMMI 评估或认证。

### 10.2 隐私提示

页面需提供简短说明：

```text
我们仅将你提交的信息用于本次项目诊断沟通，不会公开展示你的联系方式或评测结果。
```

### 10.3 隐私说明承载

V2.0-MVP 采用弹层说明，不单独建设完整隐私政策页。

弹层必须包含：

1. 收集字段：姓名、公司 / 团队、职位 / 角色、联系方式、项目背景、问卷答案和诊断结果。
2. 使用目的：项目诊断沟通、售前跟进、服务改进。
3. 保存周期：默认 180 天。
4. 删除方式：发送邮件到 `Laura_fan@126.com` 申请删除。
5. 声明：本工具不等同于官方 CMMI 评估或认证。

### 10.4 数据删除

V2.0 至少提供人工删除机制：

1. 用户可通过邮箱联系删除。
2. 内部根据联系方式或 leadId 删除对应线索。
3. 后续 V2.x 再考虑自助删除。

## 11. 非功能要求

| 类别 | 要求 |
|---|---|
| 性能 | 表单提交接口 P95 响应时间不超过 3 秒，不包含第三方邮件/飞书异步耗时 |
| 可用性 | 邮件或飞书单方失败不阻断用户查看报告 |
| 安全 | 后端需校验字段长度、格式和授权状态 |
| 反垃圾 | 同一联系方式或同一设备短时间重复提交需限流或标记 |
| 隐私 | 不在前端暴露飞书 Webhook、邮件 SMTP 密钥或后端密钥 |
| 兼容 | 现代 Chrome / Safari / Edge 可用 |
| 可追踪 | 后端记录提交、保存、邮件通知、飞书通知结果 |
| 密钥管理 | SMTP、飞书 Webhook、数据库连接均使用环境变量 |
| 日志 | 日志不得打印完整联系方式和完整项目背景 |

## 12. 埋点与业务指标

V2.0-MVP 不接入第三方统计埋点，先用后端保存的线索数据统计基础转化。

V2.0 完整版如接入统计，需在隐私说明中体现。

建议指标：

| 指标 | 含义 |
|---|---|
| questionnaire_start | 用户进入问卷 |
| questionnaire_completed | 用户完成或调整问卷 |
| lead_form_opened | 打开留资表单 |
| lead_submit_success | 留资提交成功 |
| lead_submit_failed | 留资提交失败 |
| report_unlocked | 诊断报告解锁 |

核心转化指标：

```text
问卷访问量
→ 表单打开率
→ 留资提交率
→ 售前有效线索率
→ 实际诊断预约率
```

## 13. 验收标准

### 13.1 V2.0-MVP 主流程验收

- [ ] 用户完成问卷后点击“生成项目管理自评诊断报告”。
- [ ] 页面展示留资表单。
- [ ] 桌面端为居中弹窗。
- [ ] 移动端为整页卡片。
- [ ] 必填字段未填时不能提交。
- [ ] 用户勾选授权后可提交。
- [ ] 提交成功后展示项目管理自评诊断报告。
- [ ] 后端保存线索。
- [ ] 后端保存完整问卷答案、总分、8 维得分、Top 3、授权状态和通知状态。
- [ ] 邮件发送到 `Laura_fan@126.com`。
- [ ] 邮件正文顶部包含线索摘要、建议跟进动作和建议首问问题。

### 13.2 V2.0 完整版验收

- [ ] 飞书 Webhook 已配置时，飞书收到通知。
- [ ] 飞书通知包含线索摘要、建议跟进动作和建议首问问题。
- [ ] 飞书未配置时，后端记录 `notifyStatus.feishu=not_configured`。
- [ ] 飞书发送失败时，后端记录 `notifyStatus.feishu=failed`。
- [ ] 飞书失败不阻断用户查看报告。

### 13.3 异常验收

- [ ] 后端校验失败时展示明确错误。
- [ ] 网络异常时可重试。
- [ ] 邮件失败但飞书成功时，用户仍可查看报告。
- [ ] 飞书失败但邮件成功时，用户仍可查看报告。
- [ ] 邮件和飞书都失败但线索保存成功时，用户仍可查看报告，后台记录通知失败。
- [ ] 线索保存失败时，用户不能解锁完整报告，提示重试或复制摘要联系。
- [ ] 重复点击提交不会产生多条未标记重复的线索。
- [ ] 同一联系方式 10 分钟内重复提交时，保存并标记 `duplicate=true`。
- [ ] 测试环境可以模拟邮件失败。
- [ ] 测试环境可以模拟飞书失败或未配置。

### 13.4 合规验收

- [ ] 未勾选授权不能提交。
- [ ] 页面明确说明用途为项目诊断沟通。
- [ ] 页面明确说明不等同于官方 CMMI 评估或认证。
- [ ] 前端代码不包含邮件密钥、飞书 Webhook 或后端密钥。
- [ ] 邮件和飞书内容不包含不必要的敏感信息。
- [ ] 隐私说明弹层包含收集范围、用途、保存周期和删除方式。
- [ ] 用户未勾选授权时不会调用提交接口。

## 14. 风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| 表单字段过多导致流失 | 用户不愿提交 | 字段控制在 5 个左右，职位选填 |
| 用户担心隐私 | 不愿留资 | 明确用途、授权和删除方式 |
| 邮件或飞书通知失败 | 售前无法及时跟进 | 保存线索优先，通知失败记录状态 |
| 飞书 Webhook 泄露 | 安全风险 | Webhook 只放后端环境变量 |
| 垃圾提交 | 干扰售前 | 限流、重复标记、基础校验 |
| 后端能力不足 | 无法实现自动通知 | 可先做 V2.0-MVP：保存 + 邮件，飞书作为 P1 |
| 隐私说明不足 | 用户不信任或合规风险 | 使用隐私说明弹层说明收集范围、用途、周期和删除方式 |
| 重复提交 | 线索重复干扰售前 | 保存但标记 duplicate=true |
| 测试误发邮件 | 干扰真实邮箱 | 测试环境使用 mock |
