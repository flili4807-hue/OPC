# 北京润序科技官网 V2.0 获客闭环 PRD 修订后多角色复评记录

版本：V2.0 修订后复评版  
日期：2026-07-14  
评审对象：`北京润序科技官网V2.0获客闭环PRD.md`

## 1. 复评结论

结论：通过，可以进入 V2.0 设计方案和技术方案阶段。

PRD 自检评分：90 / 100

本次修订已关闭上一轮主要 P1：

1. V2.0-MVP 与完整版边界已明确。
2. 飞书通知已定义为可配置能力，不阻断 MVP。
3. 报告解锁前后展示差异已明确。
4. 留资表单展示形态已明确为桌面弹窗、移动端整页卡片或底部抽屉。
5. 重复提交规则已明确为保存并标记 `duplicate=true`。
6. 隐私说明已明确采用弹层承载。
7. 邮件服务方案已明确为后端 SMTP，配置走环境变量。
8. 邮件 / 飞书失败测试要求已明确为 mock 或未配置状态。

仍需在技术方案阶段明确：

1. 后端具体技术栈。
2. 数据库或存储实现。
3. 生产环境部署方式。
4. SMTP 具体服务配置。
5. 飞书机器人 Webhook 配置方式。

这些属于技术方案决策，不再阻断 PRD 准入。

## 4. 分角色复评

### 4.1 业务产品经理

结论：通过。

认可点：

1. V2.0-MVP 聚焦“留资 + 邮件 + 报告解锁”，符合最短获客闭环。
2. 飞书通知不阻断 MVP，降低首版上线难度。
3. 线索质量分层能帮助后续判断售前优先级。

建议：

| 优先级 | 建议 | 说明 |
|---|---|---|
| P2 | MVP 上线后统计提交率和有效线索率 | 用真实数据判断是否继续做后台和飞书。 |

### 4.2 售前

结论：通过。

认可点：

1. 邮件内容包含客户信息、项目背景、总分、8 维度和 Top 3。
2. 增加“建议跟进动作”后，售前能更快进入沟通。
3. 高意向线索 24 小时内跟进的建议合理。

建议：

| 优先级 | 建议 | 说明 |
|---|---|---|
| P2 | 邮件标题中标注线索等级和重复状态 | 提高筛选效率。 |

### 4.3 产品负责人

结论：通过。

认可点：

1. 范围已经收敛，不再把飞书、埋点、限流全部压进 MVP。
2. 报告解锁前后状态已能指导设计。
3. 重复提交规则明确，避免研发和测试口径不一致。

建议：

| 优先级 | 建议 | 说明 |
|---|---|---|
| P2 | 技术方案中明确 MVP 排期拆分 | 便于控制上线节奏。 |

### 4.4 UI/UX 设计

结论：通过。

认可点：

1. 桌面弹窗、移动端整页卡片或底部抽屉的方向明确。
2. 解锁前隐藏完整建议，解锁后展示完整报告，有清晰转化激励。
3. 隐私弹层内容足够设计落地。

建议：

| 优先级 | 建议 | 说明 |
|---|---|---|
| P2 | 移动端优先采用整页卡片 | 字段多于普通弹窗，整页卡片更稳。 |

### 4.5 前端研发

结论：通过。

认可点：

1. 表单字段和校验规则已明确。
2. 接口请求体、成功响应和失败响应已有定义。
3. 解锁状态可以通过前端状态控制。

建议：

| 优先级 | 建议 | 说明 |
|---|---|---|
| P2 | 技术方案定义 reportUnlocked 状态机 | 包括 locked、submitting、unlocked、failed。 |

### 4.6 后端研发

结论：有条件通过。

认可点：

1. 接口职责明确。
2. 邮件默认 SMTP，飞书可配置，不再阻断 MVP。
3. 重复提交和通知状态字段明确。

进入技术方案必须明确：

1. 后端运行形态：Serverless Function / Node 服务 / 其他。
2. 存储：SQLite / PostgreSQL / 云数据库 / 表格型服务。
3. SMTP 具体服务和发送账号。
4. 飞书 Webhook 配置和失败处理。
5. 环境变量命名和密钥管理。

### 4.7 测试

结论：通过。

认可点：

1. 主流程、异常、合规验收都更具体。
2. 重复提交规则可测试。
3. 邮件 / 飞书 mock 要求解决了通知失败难测的问题。

建议：

| 优先级 | 建议 | 说明 |
|---|---|---|
| P2 | 技术方案提供测试模式参数 | 例如 `NOTIFY_MOCK=true`。 |

### 4.8 内容 / 品牌负责人

结论：通过。

认可点：

1. 授权文案、隐私说明和免责声明方向正确。
2. 明确保存周期和删除方式，有助于建立信任。
3. CMMI 表达仍保持“参考映射”，无认证误导。

建议：

| 优先级 | 建议 | 说明 |
|---|---|---|
| P2 | 隐私弹层文案后续可再润色 | 保持专业且不吓退用户。 |

### 4.9 创始人本人

结论：通过。

认可点：

1. MVP 已经能解决“客户留下来”的问题。
2. 先保存 + 邮件，飞书可配置，符合当前资源投入。
3. 1-2 个工作日响应的承诺可提升用户预期。

建议：

| 优先级 | 建议 | 说明 |
|---|---|---|
| P2 | 明确谁每天查看邮件并跟进 | 产品系统之外还需要人工运营机制。 |

### 4.10 技术负责人 / 架构

结论：有条件通过。

认可点：

1. PRD 对密钥、日志、隐私和失败状态已有约束。
2. MVP 边界合理。
3. 技术风险已经显性化。

进入技术方案必须决策：

1. 是否使用 Serverless。
2. 是否与静态官网同域部署。
3. 数据是否落本地文件、数据库或云服务。
4. 邮件发送是否使用 126 SMTP 授权码。
5. 飞书机器人是否在 MVP 启用。
