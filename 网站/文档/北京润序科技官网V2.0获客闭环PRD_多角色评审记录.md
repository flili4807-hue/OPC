# 北京润序科技官网 V2.0 获客闭环 PRD 多角色评审记录

版本：V2.0 PRD 评审版  
日期：2026-07-14  
评审对象：`北京润序科技官网V2.0获客闭环PRD.md`

## 1. 评审结论

结论：有条件通过，可以进入设计方案和技术方案阶段；开发前必须补齐 6 个 P1。

整体判断：

1. V2.0 方向正确，解决了 V1.5 最大业务缺口：没有客户信息、无法留存和跟进。
2. 从“诊断工具”升级为“获客闭环工具”符合公司官网当前商业目标。
3. PRD 已覆盖留资表单、授权、后端接口、邮件通知、飞书通知、报告解锁和异常处理。
4. 但 V2.0 涉及后端、隐私、通知服务和数据保存，复杂度显著高于 V1.5，开发前需要进一步明确技术选型、邮件服务、飞书机器人、数据存储和隐私政策落点。

## 2. 评分结论

PRD 自检评分：84 / 100

准入结论：有条件通过。

扣分原因：

1. 后端部署环境未明确。
2. 邮件发送服务未明确。
3. 飞书机器人或开放平台接入方式未明确。
4. 隐私政策页面和数据删除流程仍偏粗。
5. 埋点是否启用仍是建议态，未定最终范围。
6. V2.0-MVP 与 V2.0 完整版边界需要更强约束，否则研发排期容易膨胀。

## 4. 分角色详细评审

### 4.1 业务产品经理

结论：通过。

认可点：

1. V2.0 准确解决“看了报告但没有留下客户”的问题。
2. 报告前置留资符合 B2B 咨询型官网的获客逻辑。
3. “提交并生成报告”比“复制摘要再联系”更适合作为转化节点。

建议：

| 优先级 | 问题 | 修改要求 |
|---|---|---|
| P1 | 缺少线索质量分层 | 增加基础线索等级：高意向 / 中意向 / 待判断，可基于公司、项目背景长度、总分风险等级初步判断。 |
| P2 | 未定义转化指标目标 | 增加 V2.0 目标值，例如表单提交率、有效线索率。 |

### 4.2 售前

结论：通过。

认可点：

1. 邮件和飞书通知字段足够售前判断是否跟进。
2. 保存完整问卷答案对首次沟通有价值。
3. Top 3 短板可以直接转化为首次诊断提纲。

建议：

| 优先级 | 问题 | 修改要求 |
|---|---|---|
| P1 | 通知中缺少“建议跟进动作” | 邮件和飞书卡片增加一句售前建议，例如“建议优先确认需求变更机制和发布门禁”。 |
| P2 | 未定义线索响应 SLA | 建议定义高意向线索 24 小时内跟进。 |

### 4.3 产品负责人

结论：有条件通过。

认可点：

1. 主流程完整，覆盖用户提交、后端保存、通知和报告解锁。
2. MVP 拆分意识正确。
3. 异常状态覆盖较完整。

问题：

| 优先级 | 问题 | 修改要求 |
|---|---|---|
| P1 | V2.0-MVP 和 V2.0 完整版边界仍可能混淆 | 开发前明确第一期只做 MVP，飞书通知是否必须进入第一期需要最终确认。 |
| P1 | “完整诊断报告”定义还偏泛 | 明确报告页面具体模块、是否复用 V1.5 结果区、是否新增报告详情页。 |

### 4.4 UI/UX 设计

结论：有条件通过。

认可点：

1. “报告前留资”符合用户心理：先看到部分价值，再提交信息换取完整报告。
2. 表单字段数量相对克制。
3. 成功、失败、重试和复制摘要兜底都有定义。

问题：

| 优先级 | 问题 | 修改要求 |
|---|---|---|
| P1 | 留资表单展示形态未最终确定 | 需要明确弹窗、抽屉还是页面内卡片。建议桌面端弹窗，移动端底部抽屉或整页卡片。 |
| P1 | 报告解锁前后状态需要设计 | 需要明确未留资状态展示哪些内容、解锁后新增哪些内容。 |
| P2 | 授权文案较长 | 设计时需保证可读，不应藏在小字里。 |

### 4.5 前端研发

结论：有条件通过。

认可点：

1. 前端可复用 V1.5 的评分模型和结果对象。
2. 新增表单、校验、提交、成功/失败状态可控。
3. 报告解锁可以通过前端状态控制。

问题：

| 优先级 | 问题 | 修改要求 |
|---|---|---|
| P1 | 接口地址和环境未明确 | 需要明确开发、测试、生产接口 base URL。 |
| P1 | 表单提交防重复策略需细化 | 前端提交中禁用按钮，后端也要幂等或重复标记。 |
| P2 | 联系方式格式规则需更明确 | 手机、邮箱、飞书号至少一种，具体正则或校验策略需定义。 |

### 4.6 后端研发

结论：有条件通过。

认可点：

1. 接口字段结构基本完整。
2. 保存线索优先、通知失败不阻断用户的策略合理。
3. 不向前端暴露内部通知错误是正确的。

问题：

| 优先级 | 问题 | 修改要求 |
|---|---|---|
| P1 | 后端技术栈和部署方式未定 | 明确使用静态站配套 Serverless Function、Node 服务，还是其他后端。 |
| P1 | 邮件服务未定 | 明确 SMTP、第三方邮件服务或企业邮箱发送方式。 |
| P1 | 飞书接入方式未定 | 明确飞书机器人 Webhook 或开放平台应用。 |
| P1 | 数据库存储未定 | 明确保存到 SQLite、PostgreSQL、云数据库或表格类服务。 |
| P2 | 限流策略偏粗 | 明确同联系方式、同 IP 哈希、同设备指纹的限制口径。 |

### 4.7 测试

结论：有条件通过。

认可点：

1. 主流程和异常流程已经具备测试基础。
2. 邮件、飞书、保存线索三类结果状态拆分合理。
3. 未授权不可提交的合规验收明确。

问题：

| 优先级 | 问题 | 修改要求 |
|---|---|---|
| P1 | 邮件 / 飞书失败场景需要可模拟 | 技术方案需提供测试开关或 mock 方式。 |
| P1 | 重复提交验收需要明确数据结果 | 重复提交是拦截、覆盖还是标记重复，需要固定。 |
| P2 | 隐私删除流程缺少测试入口 | 人工删除也需要操作说明和验收方式。 |

### 4.8 内容 / 品牌负责人

结论：通过。

认可点：

1. 授权文案方向正确。
2. CMMI 参考映射表述合规。
3. “不是完整解决方案”能够降低误解。

建议：

| 优先级 | 问题 | 修改要求 |
|---|---|---|
| P1 | 隐私说明需要正式落地 | 增加隐私说明页面或弹层，至少说明收集范围、用途、保存周期、删除方式。 |
| P2 | 成功态文案应增强信任 | 增加“我们会在 1-2 个工作日内联系你”或类似承诺，需业务确认。 |

### 4.9 创始人本人

结论：通过。

认可点：

1. V2.0 对公司官网最重要：不只是展示能力，而是沉淀客户。
2. 先 MVP 验证线索质量比直接做完整后台更务实。
3. 邮件 + 飞书双通知能减少遗漏。

建议：

| 优先级 | 问题 | 修改要求 |
|---|---|---|
| P1 | 先确定 MVP 优先级 | 建议第一期必须做保存 + 邮件；飞书如果接入成本低则纳入，否则作为第二阶段。 |
| P2 | 线索跟进流程需要人工机制 | 明确谁每天检查邮件 / 飞书，如何记录跟进结果。 |

### 4.10 技术负责人 / 架构

结论：有条件通过。

认可点：

1. V2.0 引入后端是必要的，因为自动邮件和飞书不能安全放在前端。
2. 保存线索优先、通知异步处理的方向合理。
3. 密钥不暴露前端的要求正确。

问题：

| 优先级 | 问题 | 修改要求 |
|---|---|---|
| P1 | 架构选型未定 | 技术方案必须明确后端形态、部署、数据库、邮件和飞书集成。 |
| P1 | 密钥管理未定 | 邮件密码、飞书 Webhook、数据库连接必须走环境变量或密钥管理。 |
| P1 | 日志与失败补偿未定 | 至少记录保存、邮件、飞书三段状态，支持人工补偿。 |
| P2 | 数据安全策略需加强 | 联系方式建议脱敏展示，日志不要打印完整敏感信息。 |

## 5. 必须修改的问题

### P0

无。

### P1

1. `[P1][版本边界] V2.0-MVP 与完整 V2.0 边界不够强 → 影响研发排期 → 明确第一期是否包含飞书通知、埋点和限流。`
2. `[P1][架构依赖] 后端技术栈和部署方式未定 → 无法评估开发工作量 → 技术方案需明确后端形态。`
3. `[P1][通知依赖] 邮件服务和飞书接入方式未定 → 自动通知不可落地 → 明确 SMTP/邮件服务、飞书 Webhook/开放平台方案。`
4. `[P1][数据存储] 数据库或存储方案未定 → 线索无法沉淀 → 明确保存位置、字段、保存周期和删除方式。`
5. `[P1][隐私合规] 隐私说明未正式落地 → 存在用户信任和合规风险 → 增加隐私说明页面或弹层。`
6. `[P1][报告解锁] 完整诊断报告定义不够细 → 影响设计和前端实现 → 明确解锁前后展示差异。`
7. `[P1][测试可控性] 邮件 / 飞书失败场景不可模拟 → 影响验收 → 技术方案需提供 mock 或测试开关。`
8. `[P1][重复提交] 重复线索处理规则未固定 → 数据可能混乱 → 明确重复提交为标记重复，不直接丢弃。`

### P2

1. 增加线索质量分层。
2. 定义售前响应 SLA。
3. 定义 V2.0 转化目标值。
4. 明确联系方式校验规则。
5. 成功态增加预计联系时间。

## 7. 可直接写入 PRD 的补充建议

### 7.1 MVP 边界补充

```text
V2.0-MVP 第一阶段必须包含：留资表单、授权勾选、后端保存线索、邮件通知、完整报告解锁。
飞书通知如能在 1 个工作日内完成机器人配置，则纳入 MVP；否则作为 V2.0 第二阶段。
```

### 7.2 隐私说明补充

```text
我们将收集你主动提交的姓名、公司 / 团队、联系方式、项目背景和本次评测结果，仅用于北京润序科技有限责任公司进行项目诊断沟通。你可以通过 Laura_fan@126.com 联系我们删除相关信息。
```

```text
同一联系方式 10 分钟内重复提交时，系统仍保存记录，但标记为 duplicate=true，并在邮件和飞书通知中展示“重复线索”。
```

## 8. 最终准入结论

V2.0 PRD 有条件通过。

建议下一步：

1. 先按 P1 修订 PRD。
2. 然后生成 V2.0 设计方案。
3. 再生成 V2.0 技术方案，重点明确后端、邮件、飞书、数据库和密钥管理。

不建议直接进入开发，因为当前最大不确定性已经从前端体验转移到后端依赖和通知链路。
