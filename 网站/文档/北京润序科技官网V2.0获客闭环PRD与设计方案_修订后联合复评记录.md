# 北京润序科技官网 V2.0 获客闭环 PRD 与设计方案修订后联合复评记录

日期：2026-07-14  
评审对象：

- `北京润序科技官网V2.0获客闭环PRD.md`
- `北京润序科技官网V2.0获客闭环设计方案.md`

评审背景：

本次评审针对上一轮联合评审 P1 修订后的 PRD 与设计方案进行复评，重点确认产品口径、设计口径、前后端实现边界、验收标准和上线风险是否已经对齐。

评审角色：

- 业务产品经理
- 售前
- 产品负责人
- UI/UX 设计
- 前端研发
- 后端研发
- 测试
- 内容 / 品牌负责人
- 创始人本人
- 技术负责人 / 架构

## 1. 总体结论

结论：通过，可进入 V2.0 技术方案评审与开发准备。

修订后已经解决上一轮 P1 的核心问题：

1. 移动端表单形态已统一为“整页卡片”。
2. 报告命名已统一为“项目管理自评诊断报告”。
3. 解锁条件已统一为“服务端校验通过 + 线索保存成功”。
4. 邮件 / 飞书通知失败不阻断报告解锁。
5. 飞书已定义为可配置能力，未配置时记录 `not_configured`。
6. 售前通知已增加线索摘要、建议跟进动作和建议首问问题。
7. 数据保存介质已明确由技术方案确认，优先 SQLite 或 PostgreSQL。
8. 页面会话解锁规则已明确，刷新后回到未解锁状态。

当前 PRD 与设计方案已经具备进入技术方案和研发实现的条件。

## 2. 分角色复评意见

### 2.1 业务产品经理

结论：通过。

通过点：

- 获客链路已经闭合：问卷 → 留资 → 保存线索 → 通知 → 解锁报告。
- 解锁条件从“通知成功”调整为“线索保存成功”，业务闭环更稳定。
- 页面会话解锁符合 MVP 阶段，不引入账号体系和长期凭证。

保留建议：

- P2：后续可增加线索来源细分，如首页 CTA、问卷页 CTA、联系区 CTA。

### 2.2 售前

结论：通过。

通过点：

- 邮件和飞书内容已要求包含线索摘要、建议跟进动作和建议首问问题。
- 线索等级 high / medium / unknown 足够支持早期人工跟进排序。
- 项目背景占位文案能引导用户说明真实问题。

保留建议：

- P2：邮件正文顶部建议固定顺序为“联系方式 → 公司 → 分数 → 风险等级 → 建议首问问题”，方便手机端快速查看。

### 2.3 产品负责人

结论：通过。

通过点：

- MVP 与完整版验收已拆分。
- 飞书能力不再阻断 MVP 验收。
- PRD 与设计方案关键路径一致。

保留建议：

- P2：V2.0-MVP 版本命名建议在开发任务中明确，避免把飞书完整通知、埋点、补偿机制混入第一期。

### 2.4 UI/UX 设计

结论：通过。

通过点：

- 解锁前 / 解锁后信息差明确。
- 留资表单层级清晰。
- 移动端使用整页卡片，避免小屏弹窗输入体验差。
- CTA 和报告命名已统一为“项目管理自评诊断报告”。

保留建议：

- P2：成功提示文案建议同步改为“项目管理自评诊断报告已生成”，当前设计方案中仍有一处“诊断报告已生成”的短表述，开发实现时需要使用 PRD 的完整表述。

### 2.5 前端研发

结论：通过。

通过点：

- 状态机清晰：`locked`、`form_open`、`submitting`、`unlocked`、`failed`。
- 解锁和失败状态定义明确。
- 表单关闭后保留内容、刷新后回到未解锁，前端实现边界可控。

保留建议：

- P1：技术方案和开发实现中必须明确表单临时数据存储方式，建议使用页面内存或 `sessionStorage`，不使用 `localStorage`。
- P2：提交失败后需要保留滚动位置，避免移动端用户丢失上下文。

### 2.6 后端研发

结论：通过。

通过点：

- 接口成功标准明确为“服务端校验通过 + 线索保存成功”。
- 邮件 / 飞书失败通过 `notifyStatus` 记录，不阻断用户。
- 数据保存字段、重复标记、通知状态、线索质量均已定义。

保留建议：

- P1：技术方案中必须明确数据库最终选型和部署环境持久化能力。
- P1：错误码需要落到技术方案和接口实现中，至少包含 `VALIDATION_ERROR`、`CONSENT_REQUIRED`、`RATE_LIMITED`、`SAVE_FAILED`、`UNKNOWN_ERROR`。

### 2.7 测试

结论：通过。

通过点：

- MVP 主流程、完整版飞书验收、异常验收、合规验收已经拆分。
- 邮件失败、飞书失败、保存失败、重复提交都有验收口径。
- 移动端、桌面端、刷新后状态都有明确检查项。

保留建议：

- P1：技术方案需补充联调测试矩阵，覆盖保存成功/失败、邮件成功/失败、飞书成功/失败/未配置。
- P2：增加 375px、768px、1440px 三档截图验收。

### 2.8 内容 / 品牌负责人

结论：通过。

通过点：

- “完整诊断报告”已调整为“项目管理自评诊断报告”，降低过度承诺风险。
- CMMI 表达保持“参考映射”，未出现官方认证或认证结果。
- 隐私说明、保存周期和删除方式均已覆盖。

保留建议：

- P2：所有成功提示、按钮和邮件标题中避免“正式报告”“认证结果”等表述。

### 2.9 创始人本人

结论：通过。

通过点：

- 能解决官网诊断后无法留资的问题。
- 早期不做复杂后台，但要求后端保存可查询数据，符合 MVP。
- 邮件通知到 `Laura_fan@126.com`，适合早期人工跟进。

保留建议：

- P1：上线前必须确认线索数据可人工导出或至少可人工查询，不允许只有邮件通知。

### 2.10 技术负责人 / 架构

结论：通过。

通过点：

- 产品和设计已经具备技术方案输入条件。
- 关键架构约束明确：后端接口、数据保存、邮件、飞书可配置、密钥后端环境变量。
- 不引入账号、CRM、PDF、日历，符合第一期复杂度控制。

保留建议：

- P1：技术方案必须确认生产部署方式，尤其是静态站点如何挂载后端 API。
- P1：如果选择 SQLite，必须确认生产环境有持久化磁盘；否则直接使用 PostgreSQL。

## 3. 复评问题清单
