# 北京润序科技官网 V2.0 获客闭环 PRD 与设计方案联合评审记录

## 1. 总体结论

结论：有条件通过。

当前 V2.0 方向正确：从“问卷诊断工具”升级为“获客闭环工具”，核心动线已经成立：

```text
完成问卷
→ 查看基础诊断结果
→ 点击生成完整报告
→ 提交咨询者信息
→ 后端保存线索
→ 邮件 / 飞书通知
→ 解锁完整报告
```

PRD 和设计方案已覆盖主要业务目标、用户路径、字段、隐私授权、前后端状态和基础验收标准，可以进入技术方案阶段。

但进入开发前仍需修正几处 P1 问题，主要集中在：

1. PRD 与设计对移动端表单形态表述不一致。
2. 飞书通知在 MVP 中是可配置能力，但验收标准写成了必须收到。
3. 邮件 / 飞书失败时是否解锁报告的规则需要再统一。
4. “完整报告解锁”的前端展示边界需要和后端保存成功条件绑定。
5. 售前收到的通知内容需要更偏向行动判断，而不只是数据转发。

## 2. 分角色评审意见

### 2.1 业务产品经理

通过点：

- 获客动线从 V1.5 的“用户主动联系”升级为 V2.0 的“系统主动沉淀线索”，业务目标明确。
- 留资发生在用户完成问卷并看到基础结果之后，转化时机合理。
- 线索字段包含姓名、公司、联系方式、项目背景和诊断结果，满足售前初筛。

问题：

- PRD 的“提交成功后完整报告解锁”需要明确：是“线索保存成功即解锁”，还是“邮件发送成功才解锁”。
- 当前线索质量分层主要基于项目背景长度和分数，缺少“是否有明确咨询诉求”的判断。

建议：

- P1：明确解锁条件为“后端保存线索成功”，邮件 / 飞书通知失败不阻断解锁，但需要记录通知失败状态。
- P2：在项目背景字段中引导用户补充“当前最想解决的问题”，便于售前判断意向。

### 2.2 售前

通过点：

- 邮件内容包含客户信息、分数、8 维得分、Top 3 短板和建议，具备跟进基础。
- 线索等级 high / medium / unknown 有助于初步排序。

问题：

- 通知内容偏“报告转发”，还需要突出“下一步沟通建议”。
- 邮件标题建议增加风险等级，方便收件箱快速筛选。

建议：

- P1：邮件 / 飞书通知必须包含“建议首问问题”或“建议跟进动作”。
- P2：邮件标题增加风险等级，例如：`【润序科技官网线索｜高风险】{{公司}} - {{姓名}} - {{总分}}分`。

### 2.3 产品负责人

通过点：

- V2.0-MVP 和完整版已拆开，避免第一阶段过度建设。
- 明确了不做 CRM、不做后台、不做 PDF、不做日历预约，范围控制合理。

问题：

- PRD 中移动端表单仍写“整页卡片或底部抽屉”，而设计方案已明确为“整页卡片”，需要统一。
- PRD 验收标准写“飞书收到通知”，但 PRD 同时定义 MVP 中飞书是可配置能力，存在验收冲突。

建议：

- P1：PRD 中移动端表单统一为“整页卡片”。
- P1：验收标准区分 MVP 与完整版：MVP 必验邮件，飞书若未配置则验 `not_configured` 状态；完整版再验飞书通知。

### 2.4 UI/UX 设计

通过点：

- 设计方案已补齐解锁前 / 后差异，用户能理解为什么要留资。
- 留资弹窗层级已清晰：标题、价值说明、字段、授权、隐私入口、按钮。
- 移动端统一整页卡片，输入体验优于弹窗或抽屉。

问题：

- 完整报告价值说明还可以更贴近业务用户语言，不只是“完整治理建议”，还要强调“可带着结果预约诊断”。
- 解锁前 Top 3 只展示名称和分数是合理的，但需要避免让用户误解“报告已经完整”。

建议：

- P1：解锁前 CTA 附近增加一句短文案：`提交咨询信息后，可查看 Top 3 治理建议，并带着报告预约项目诊断。`
- P2：完整报告区域增加“适合项目经理 / 创始人用于首次诊断沟通”的说明。

### 2.5 前端研发

通过点：

- 设计方案已定义 `locked / form_open / submitting / unlocked / failed` 状态，前端实现边界清晰。
- 解锁前后展示差异清楚，适合做条件渲染。

问题：

- 需要明确表单内容关闭后保存在哪里：页面内状态、sessionStorage，还是 localStorage。
- 需要明确提交成功后刷新页面是否仍保持解锁。当前方案更像一次性会话解锁，但未明确。

建议：

- P1：本期按“页面会话解锁”处理，刷新页面后回到未解锁状态，避免引入用户身份和长期凭证。
- P1：关闭表单后的已填写内容保存在前端内存状态或 sessionStorage，不使用 localStorage。
- P2：接口调用失败时保留问卷答案、表单内容和当前滚动位置。

### 2.6 后端研发

通过点：

- PRD 已定义提交接口、请求体、响应体、数据对象和通知状态。
- SMTP、飞书 Webhook、数据库连接使用环境变量，安全边界正确。

问题：

- 需要明确 MVP 数据保存介质：文件、SQLite、轻量数据库或云数据库。
- 需要明确邮件发送失败时的事务边界：线索已保存但通知失败，应返回成功还是失败。
- 需要明确限流维度：联系方式、设备哈希、IP 哈希还是组合维度。

建议：

- P1：技术方案中明确保存策略；MVP 可用 SQLite 或 JSONL，但生产建议 SQLite / PostgreSQL，不建议长期只用 JSON 文件。
- P1：接口成功标准定义为“服务端校验通过 + 线索保存成功”。通知失败不影响用户解锁，但写入 `notifyStatus`。
- P1：限流至少包含联系方式哈希 + IP 哈希，日志不得打印完整联系方式。

### 2.7 测试

通过点：

- PRD 和设计都已经列出主流程、异常流程、响应式和合规验收。
- 表单校验规则足够生成测试用例。

问题：

- 验收标准中“飞书收到通知”与“飞书可配置”冲突，会导致测试口径不一致。
- 邮件 mock、飞书 mock、后端保存失败、通知失败的测试数据需要提前定义。

建议：

- P1：拆分测试矩阵：MVP 必测、完整版必测、可配置能力测试。
- P1：补充 6 类异常用例：字段校验失败、网络失败、保存失败、邮件失败、飞书失败、重复提交。
- P2：增加 375px、768px、1440px 三档截图验收。

### 2.8 内容 / 品牌负责人

通过点：

- 文案整体克制，没有夸大 CMMI 认证或官方评估。
- 隐私说明和免责声明方向正确。

问题：

- “项目成功率得分”可能被用户理解为精确预测，需要加“自评诊断”语境。
- “完整诊断报告”不要暗示正式咨询交付件，应明确是“自评诊断报告”。

建议：

- P1：页面统一使用“项目成功率自评诊断报告”或“项目管理自评诊断报告”。
- P1：所有 CMMI 相关表达保持“参考映射”，不能出现“认证结果”“官方评估”。
- P2：成功提示中补充“我们会基于你提交的信息进行初步判断”，避免承诺一定提供完整方案。

### 2.9 创始人本人

通过点：

- V2.0 能解决当前最关键问题：官网有人看、有人测，但无法留下客户信息。
- 邮件发送到 `Laura_fan@126.com`，适合作为早期轻量线索承接。
- 解锁完整报告前留资，符合咨询业务的获客逻辑。

问题：

- 早期不需要复杂后台，但必须能可靠查到线索。
- 通知内容必须能一眼判断是否值得跟进。

建议：

- P1：邮件正文顶部放“线索摘要”：公司、姓名、联系方式、分数、风险等级、线索等级、建议跟进动作。
- P1：后端保存的数据必须可导出或可人工查看，避免只有邮件通知。

### 2.10 技术负责人 / 架构

通过点：

- V2.0 范围控制合理，先做获客闭环，不引入账号、CRM、PDF、日历等复杂能力。
- 前端状态、接口、通知状态、隐私要求已经具备技术方案输入条件。

问题：

- 生产部署形态尚未确定：静态页面 + Serverless API、Node 服务、还是其他后端。
- 如果继续使用本地静态服务，无法完成真实邮件和飞书通知。
- 需要明确环境变量、日志脱敏、错误码和部署回滚方案。

建议：

- P1：技术方案必须明确部署架构，至少包括前端静态站点、后端 API、数据存储、邮件服务、飞书通知、环境变量。
- P1：定义错误码：`VALIDATION_ERROR`、`SAVE_FAILED`、`RATE_LIMITED`、`NOTIFY_FAILED`、`UNKNOWN_ERROR`。
- P2：补充最小运维方案：日志位置、数据备份、密钥轮换、手动导出线索方式。

## 4. P2 优化项

| 编号 | 建议 | 价值 |
|---|---|---|
| P2-01 | 项目背景增加占位提示：当前阶段、主要问题、希望诊断方向 | 提高线索质量 |
| P2-02 | 邮件标题增加风险等级 | 提高收件箱筛选效率 |
| P2-03 | 增加 375px / 768px / 1440px 截图验收 | 降低响应式返工 |
| P2-04 | 成功提示弱化承诺，不承诺一定输出完整方案 | 降低用户预期风险 |
| P2-05 | 后续增加线索 CSV 导出 | 便于早期人工运营 |

## 5. 修订建议优先级

建议先修 PRD，再修设计方案，最后进入技术方案。

顺序：

1. 修订 PRD 中 P1-01、P1-02、P1-03、P1-06、P1-07。
2. 同步修订设计方案中的报告命名、CTA 文案和解锁说明。
3. 生成 V2.0 技术方案，重点确认后端架构、数据保存、邮件 / 飞书、错误码、日志脱敏。
4. 技术方案评审通过后再进入开发。

## 6. 最终评审结论

当前 PRD 与设计方案具备进入下一步的基础，但不建议直接开发。

通过条件：

1. P1 项完成修订。
2. PRD、设计方案、技术方案三者口径统一。
3. 技术方案明确真实可上线的后端、存储、邮件、飞书与异常处理方案。

评审结论：有条件通过，完成 P1 修订后可进入 V2.0 技术方案。
