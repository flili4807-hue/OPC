## 1. 技术结论

V2.0 从 V1.5 的纯静态问卷升级为“前端问卷 + 后端线索接口 + 数据保存 + 邮件通知 + 飞书可配置通知”的获客闭环。

推荐方案：

```text
静态官网页面
→ 前端本地计算问卷结果
→ 用户点击生成项目管理自评诊断报告
→ 留资表单提交
→ 后端 API 校验并保存线索
→ 邮件发送到 Laura_fan@126.com
→ 飞书 Webhook 可配置发送
→ 前端解锁项目管理自评诊断报告
```

V2.0-MVP 技术决策：

| 事项 | 决策 |
|---|---|
| 部署方式 | Node.js 服务同时托管静态文件和 `/api` |
| 数据库 | SQLite，数据库文件放在后端数据目录 |
| 后续扩展 | 若迁移到 Serverless 或正式云部署，改 PostgreSQL |
| 本地联调 | 同一 Node 服务提供页面和接口，避免跨域 |
| 表单临时数据 | `sessionStorage`，不使用 `localStorage` |
| 线索导出 | 提供 CSV 导出脚本 |
| 邮件通知 | SMTP；测试环境 `EMAIL_MOCK=true` |
| 飞书通知 | 可配置；未配置返回 `not_configured` |
| 通知超时 | 邮件 / 飞书单项 3 秒超时 |
| 前端接口超时 | 8-10 秒超时 |

MVP 成功标准：

1. 线索保存成功。
2. 邮件可发送或测试环境可 mock。
3. 前端能基于接口成功响应解锁报告。
4. 飞书未配置不阻断上线。

## 2. 架构方案

### 2.1 总体架构

```text
Browser
├─ index.html
├─ questionnaire.html
├─ styles.css
└─ questionnaire.js
   ↓ POST /api/leads/diagnostic

Backend API
├─ 字段校验
├─ 诊断结果校验
├─ 重复线索识别
├─ 线索质量分层
├─ SQLite / PostgreSQL 保存
├─ SMTP 邮件通知
└─ 飞书 Webhook 通知，可配置
```

### 2.2 部署建议

MVP 推荐：

| 层级 | 方案 | 说明 |
|---|---|---|
| 前端 | Node.js 托管静态文件 | 保持当前 HTML / CSS / JS 结构，由同一服务输出 |
| 后端 | Node.js API 服务 | 同一服务提供 `/api/leads/diagnostic` |
| 存储 | SQLite | 早期轻量、可备份、可人工导出 CSV |
| 邮件 | SMTP | 使用 126 邮箱授权码或独立 SMTP 服务 |
| 飞书 | Webhook 可配置 | 未配置时记录 `not_configured` |

生产扩展建议：

1. 如果线索量增加，SQLite 可迁移到 PostgreSQL。
2. 如果部署到 Serverless，需确认 SQLite 持久化能力；否则直接用 PostgreSQL。
3. 邮件和飞书通知 MVP 可先同步执行，但单项通知必须设置 3 秒超时。
4. 前端和后端本地联调使用同一 Node 服务，不单独配置跨域。

## 4. 前端技术方案

### 4.1 文件影响范围

| 文件 | 改造内容 |
|---|---|
| `questionnaire.html` | 增加留资表单、隐私弹层、解锁后报告区域 |
| `questionnaire.js` | 增加状态机、表单校验、接口提交、报告解锁 |
| `styles.css` | 增加弹窗、整页卡片、表单、成功失败态样式 |
| `index.html` | 联系区文案可同步承接自评报告 |

### 4.2 前端状态机

| 状态 | 触发 | 页面表现 |
|---|---|---|
| `locked` | 默认状态 | 展示基础诊断、雷达图、Top 3 名称和分数 |
| `form_open` | 点击 CTA | 展示留资表单 |
| `submitting` | 点击提交 | 禁用按钮，显示提交中 |
| `unlocked` | 接口返回 `success=true` 且 `reportUnlocked=true` | 展示项目管理自评诊断报告 |
| `failed` | 接口失败或网络失败 | 展示错误，保留表单内容 |

刷新规则：

1. 本期只做页面会话解锁。
2. 刷新页面后回到 `locked`。
3. 表单关闭后，已填写内容保存在 `sessionStorage`，不使用 `localStorage`。
4. 刷新页面后清空报告解锁状态，表单临时数据可清空。

### 4.3 解锁前后展示

解锁前：

1. 总分。
2. 风险等级。
3. CMMI 参考映射。
4. 雷达图。
5. 8 维度分数。
6. Top 3 维度名称和分数。
7. CTA：`生成项目管理自评诊断报告`。

解锁后：

1. 完整 Top 3 治理建议。
2. 下一步诊断建议。
3. 报告摘要复制。
4. 成功提示。
5. 免责声明。

### 4.4 前端校验

前端必须校验：

| 字段 | 规则 |
|---|---|
| 姓名 | 2-20 字 |
| 公司 / 团队 | 2-50 字 |
| 职位 / 角色 | 0-30 字 |
| 联系方式 | 手机 / 邮箱 / 飞书号至少一种有效 |
| 项目背景 | 10-500 字 |
| 授权 | 必须勾选 |

说明：前端校验只用于体验，后端必须重复校验。

### 4.5 提交请求

```js
fetch('/api/leads/diagnostic', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    profile,
    diagnostic,
    clientMeta
  })
});
```

前端处理规则：

1. `success=true && reportUnlocked=true`：进入 `unlocked`。
2. `VALIDATION_ERROR`：展示字段错误或顶部错误。
3. `SAVE_FAILED`：不解锁，提示重试或复制摘要联系。
4. 网络失败：进入 `failed`，保留表单内容。
5. 不展示内部邮件或飞书错误详情。
6. 接口请求超时建议 8-10 秒；超时进入 `failed`。

## 5. 后端技术方案

### 5.1 API

```text
POST /api/leads/diagnostic
```

请求体：

```json
{
  "profile": {
    "name": "张三",
    "company": "某科技公司",
    "role": "产品负责人",
    "contact": "zhangsan@example.com",
    "projectBackground": "当前项目跨团队协作复杂，需求变更频繁，版本延期明显。",
    "consent": true
  },
  "diagnostic": {
    "score": 42,
    "level": "高风险",
    "cmmiReference": "参考映射：CMMI L1-L2",
    "dimensions": [],
    "weaknesses": [],
    "answers": {}
  },
  "clientMeta": {
    "page": "questionnaire.html",
    "userAgent": "browser",
    "submittedAt": "2026-07-14T16:24:00+08:00"
  }
}
```

成功响应：

```json
{
  "success": true,
  "leadId": "lead_20260714_xxxxxx",
  "reportUnlocked": true,
  "duplicate": false,
  "notifyStatus": {
    "email": "success",
    "feishu": "not_configured"
  }
}
```

失败响应：

```json
{
  "success": false,
  "code": "VALIDATION_ERROR",
  "message": "请填写有效联系方式"
}
```

### 5.2 服务端处理流程

```text
接收请求
→ 校验字段
→ 校验 consent=true
→ 校验诊断结果结构
→ 生成 leadId
→ 计算 contactHash / ipHash
→ 判断 duplicate
→ 计算 leadQuality
→ 保存线索
→ 发送邮件，3 秒超时
→ 如配置飞书则发送飞书，3 秒超时
→ 返回 reportUnlocked=true
```

关键规则：

1. 线索保存成功是解锁报告的必要条件。
2. 邮件 / 飞书失败不阻断报告解锁。
3. 邮件 / 飞书失败必须记录 `notifyStatus`。
4. 线索保存失败时返回 `SAVE_FAILED`，前端不解锁。
5. 通知超时按失败处理，记录状态，不阻断解锁。

### 5.3 错误码

| 错误码 | 场景 | 前端表现 |
|---|---|---|
| `VALIDATION_ERROR` | 字段不合法 | 展示明确错误 |
| `CONSENT_REQUIRED` | 未授权 | 提示勾选授权 |
| `RATE_LIMITED` | 高频提交 | 提示稍后再试 |
| `SAVE_FAILED` | 线索保存失败 | 不解锁，提示重试 |
| `NOTIFY_FAILED` | 通知失败但保存成功 | 不返回给前端作为失败，仅写入日志 |
| `UNKNOWN_ERROR` | 未知异常 | 统一失败提示 |

## 6. 数据存储方案

### 6.1 MVP 存储选择

V2.0-MVP 锁定 SQLite。

原因：

1. 早期部署简单。
2. 支持结构化查询。
3. 可备份、可导出。
4. 比纯 JSON 文件更稳定。

如果部署环境不支持持久化文件，直接改用 PostgreSQL；该调整需要进入后续部署方案，不阻塞本地 MVP 开发。

### 6.2 表结构

表名：`diagnostic_leads`

| 字段 | 类型 | 说明 |
|---|---|---|
| `lead_id` | text primary key | 线索 ID |
| `created_at` | text | 创建时间 |
| `source` | text | 来源 |
| `name` | text | 姓名 |
| `company` | text | 公司 / 团队 |
| `role` | text | 职位 / 角色 |
| `contact` | text | 联系方式，明文保存用于跟进 |
| `contact_hash` | text | 联系方式哈希，用于重复判断 |
| `project_background` | text | 项目背景 |
| `consent` | integer | 是否授权 |
| `score` | integer | 总分 |
| `level` | text | 风险等级 |
| `cmmi_reference` | text | CMMI 参考映射 |
| `dimensions_json` | text | 8 维得分 JSON |
| `weaknesses_json` | text | Top 3 JSON |
| `answers_json` | text | 24 题答案 JSON |
| `lead_quality` | text | high / medium / unknown |
| `duplicate` | integer | 是否重复 |
| `notify_email` | text | success / failed / mock |
| `notify_feishu` | text | success / failed / not_configured / mock |
| `ip_hash` | text | IP 哈希 |
| `user_agent_hash` | text | UA 哈希 |

索引：

```sql
CREATE INDEX IF NOT EXISTS idx_diagnostic_leads_created_at ON diagnostic_leads(created_at);
CREATE INDEX IF NOT EXISTS idx_diagnostic_leads_contact_hash ON diagnostic_leads(contact_hash);
CREATE INDEX IF NOT EXISTS idx_diagnostic_leads_duplicate ON diagnostic_leads(duplicate);
CREATE INDEX IF NOT EXISTS idx_diagnostic_leads_quality ON diagnostic_leads(lead_quality);
```

### 6.3 CSV 导出

MVP 提供脚本：

```text

```

导出内容：

1. leadId。
2. createdAt。
3. company。
4. name。
5. role。
6. contact。
7. score。
8. level。
9. leadQuality。
10. duplicate。
11. notifyEmail。
12. notifyFeishu。
13. projectBackground。

导出文件：

```text

```

### 6.4 数据保留

1. 默认保留 180 天。
2. V2.0 不做自动删除任务，但需支持人工按联系方式或 leadId 删除。
3. 后续 V2.x 增加定时清理。

### 6.5 人工删除

MVP 可先提供脚本级删除能力：

```text
node scripts/delete-lead.js --lead-id lead_xxx
node scripts/delete-lead.js --contact xxx@example.com
```

删除动作必须记录操作时间和删除条件，不在普通日志中输出完整联系方式。

## 8. 飞书通知方案

### 8.1 配置

环境变量：

```text
FEISHU_WEBHOOK_URL=
FEISHU_MOCK=false
FEISHU_TIMEOUT_MS=3000
```

规则：

1. `FEISHU_WEBHOOK_URL` 为空时，记录 `notify_feishu=not_configured`。
2. 已配置时发送飞书卡片。
3. 发送失败时记录 `notify_feishu=failed`。
4. 飞书失败不阻断报告解锁。
5. 飞书发送单次超时 3 秒，超时记录 `notify_feishu=failed`。

### 8.2 飞书卡片内容

必须包含：

1. 线索摘要。
2. 建议跟进动作。
3. 建议首问问题。
4. 姓名、公司、联系方式。
5. 项目背景。
6. 总分、风险等级、CMMI 参考映射。
7. Top 3 短板。

## 9. 安全与隐私

### 9.1 密钥管理

不得进入前端代码：

1. SMTP 用户名。
2. SMTP 密码或授权码。
3. 飞书 Webhook。
4. 数据库连接串。
5. 任意后端密钥。

### 9.2 日志脱敏

日志允许记录：

1. leadId。
2. contactHash。
3. ipHash。
4. score。
5. level。
6. notifyStatus。

日志禁止记录：

1. 完整联系方式。
2. 完整项目背景。
3. SMTP 密钥。
4. 飞书 Webhook。

### 9.3 限流

MVP 基础策略：

1. 同一联系方式哈希 10 分钟内重复提交，保存但标记 `duplicate=true`。
2. 同一 IP 哈希短时间大量提交，返回 `RATE_LIMITED`。
3. 具体阈值建议：10 分钟 5 次。

### 9.4 环境变量样例

项目必须提供 `.env.example`：

```text
PORT=8765
NODE_ENV=development
DATABASE_PATH=./data/runxu-leads.sqlite
LEAD_NOTIFY_EMAIL=Laura_fan@126.com
SMTP_HOST=
SMTP_PORT=465
SMTP_SECURE=true
SMTP_USER=
SMTP_PASS=
EMAIL_MOCK=true
MAIL_TIMEOUT_MS=3000
FEISHU_WEBHOOK_URL=
FEISHU_MOCK=true
FEISHU_TIMEOUT_MS=3000
RATE_LIMIT_WINDOW_MS=600000
RATE_LIMIT_MAX=5
```

### 9.5 安全检查

上线前必须检查：

1. 前端源码不包含 `SMTP_PASS`、`FEISHU_WEBHOOK_URL`、邮箱授权码。
2. 日志不打印完整联系方式。
3. 日志不打印完整项目背景。
4. `.env` 不提交到 Git。
5. `data/` 下数据库和导出文件不提交到 Git。

5. 邮件真实发送。
6. 前后端联调。
7. 飞书可配置通知。
8. 响应式和异常测试。
9. 上线前安全检查。

## 13. 技术风险

| 风险 | 影响 | 应对 |
|---|---|---|
| 本地静态服务无法发送邮件 | 无法完成获客闭环 | 必须增加后端 API |
| SQLite 持久化受部署环境限制 | 数据丢失 | 部署前确认持久化；不支持则用 PostgreSQL |
| SMTP 授权码配置失败 | 邮件不达 | 支持 mock 和失败状态记录 |
| 飞书 Webhook 泄露 | 安全风险 | 仅后端环境变量保存 |
| 通知慢导致接口超时 | 用户等待长 | 设置通知超时；后续改异步队列 |
| 垃圾提交 | 干扰售前 | 字段校验、重复标记、限流 |

## 14. 结论

V2.0 技术方案可进入开发。

开发前必须确认：

1. 生产或本地预览环境可以运行 Node.js 服务。
2. SQLite 数据库目录可写且可持久化。
3. SMTP 账号和授权码是否可用。
4. 飞书 Webhook 是否本期配置。
5. CSV 导出脚本可用。

若以上条件暂未全部具备，仍可先开发前端状态机、后端接口、SQLite 保存和邮件 mock，保证主链路可联调。
