# 北京润序科技官网 V2.0 获客闭环技术方案多角色评审记录

日期：2026-07-14  
评审对象：`北京润序科技官网V2.0获客闭环技术方案.md`  
关联 PRD：`北京润序科技官网V2.0获客闭环PRD.md`  
关联设计：`北京润序科技官网V2.0获客闭环设计方案.md`

评审角色：

- 业务产品经理
- 售前
- 产品负责人
- UI/UX 设计
- 前端研发
- 后端研发
- 测试

- 创始人本人
- 技术负责人 / 架构

## 1. 总体结论

结论：有条件通过。

技术方案已经覆盖 V2.0 MVP 的关键能力：

1. 前端留资表单。
2. 前端状态机。
3. 后端线索提交接口。
4. 服务端校验。
5. 线索保存。
6. 邮件通知。
7. 飞书可配置通知。
8. 日志脱敏。
9. 测试环境 mock。
10. 联调测试矩阵。

方案方向可行，可以进入开发准备。

但进入开发前必须锁定以下 P1 项：

1. 生产部署方式。
2. 数据库最终选型和持久化策略。
3. 线索人工查询 / 导出方式。
4. SMTP 账号与授权码可用性。
5. 通知超时策略和接口响应时限。
6. 前后端本地联调方式。

## 2. 分角色评审意见

### 2.1 业务产品经理

结论：通过，有 P1 前置确认项。

通过点：

- 技术方案能承接 PRD 的核心目标：留资、保存、通知、解锁报告。
- “线索保存成功即解锁”的规则已落到技术流程。
- 邮件 / 飞书失败不阻断用户体验，符合获客转化优先级。

问题：

- 当前技术方案没有明确“线索如何被业务侧查看”。如果只有数据库，不利于早期跟进。

建议：

- P1：MVP 至少提供一种人工查看方式：CSV 导出脚本、数据库查询说明或管理者可读的线索文件导出。
- P2：后续增加线索来源字段细分，支持分析不同入口转化。

### 2.2 售前

结论：通过。

通过点：

- 邮件正文顶部包含线索摘要、建议跟进动作、建议首问问题，满足售前快速判断。
- 风险等级、总分、Top 3 短板和项目背景均会进入邮件。
- 重复线索会被标记，降低重复跟进成本。

问题：

- 如果邮件失败但线索保存成功，售前不会及时知道，必须有可查询兜底。

建议：

- P1：通知失败的线索需要能通过人工查询或导出发现。
- P2：邮件标题建议保留“风险等级 + 公司 + 姓名 + 总分”，方便搜索。

### 2.3 产品负责人

结论：通过。

通过点：

- 技术范围没有扩散到登录、CRM、PDF、日历，符合 MVP。
- 飞书作为可配置能力，不阻断第一期上线。
- 错误码、状态机、验收矩阵比较完整。

问题：

- “SQLite 或 PostgreSQL”仍是二选一，需要在开发前确认，否则会影响后端实现。

建议：

- P1：根据最终部署环境锁定数据库方案；如果使用本机 / VPS，可先 SQLite；如果使用 Serverless，优先 PostgreSQL。

### 2.4 UI/UX 设计

结论：通过。

通过点：

- 前端状态机能支撑设计中的交互状态。
- 移动端整页卡片、提交中、失败保留内容等规则可以实现。
- 解锁前后展示差异清晰。

问题：

- 技术方案没有明确提交失败后是否回到表单位置、是否保留滚动位置。

建议：

- P2：移动端提交失败后保留表单内容和当前位置，避免用户重新滚动。
- P2：`failed` 状态优先在表单顶部和按钮附近同时提示。

### 2.5 前端研发

结论：有条件通过。

通过点：

- 文件影响范围明确。
- 状态机明确。
- 接口成功 / 失败处理明确。
- 不使用 `localStorage` 的原则合理。

问题：

- 未明确本地开发时前端静态服务如何代理 `/api/leads/diagnostic`。
- 未明确表单临时数据最终选用页面内存还是 `sessionStorage`。
- 未明确接口超时后前端如何处理。

建议：

- P1：本地开发需要确定前端服务与后端 API 的联调方式，例如同一个 Node 服务托管静态文件和 API，或配置本地代理。
- P1：表单临时数据建议使用 `sessionStorage`，关闭表单后恢复，刷新后可清空或按产品规则清空。
- P1：前端接口请求建议设置 8-10 秒超时，超时进入 `failed`。

### 2.6 后端研发

结论：有条件通过。

通过点：

- API、请求体、响应体、错误码、数据表结构已经具备实现基础。
- 通知失败不阻断报告解锁的事务边界清晰。
- 日志脱敏和密钥环境变量要求明确。

问题：

- SQLite 表缺少部分索引定义，如 `created_at`、`contact_hash`。
- 通知同步发送可能拖慢接口响应。
- 缺少数据导出和人工删除的最小实现方式。

建议：

- P1：增加索引：`contact_hash`、`created_at`、`duplicate`、`lead_quality`。
- P1：邮件 / 飞书发送设置超时，建议单个通知 3 秒内结束；后续再做异步队列。
- P1：提供最小 CSV 导出脚本或只读导出接口，仅限本地/管理员使用。
- P2：人工删除可以先提供脚本，根据 `lead_id` 或 `contact_hash` 删除。

### 2.7 测试

结论：有条件通过。

通过点：

- 已包含前端测试、后端测试和联调矩阵。
- 覆盖保存失败、邮件失败、飞书失败、字段失败等关键异常。

问题：

- 测试环境配置项还不够明确，比如 `EMAIL_MOCK`、`FEISHU_MOCK`、数据库路径如何区分。
- 缺少安全检查用例，例如前端是否包含密钥、日志是否打印敏感数据。

建议：

- P1：增加环境配置检查清单：开发、测试、生产三套环境变量。
- P1：增加安全测试：前端源码无 SMTP / Webhook、日志无完整联系方式和完整项目背景。
- P2：补充 375px、768px、1440px 的截图验收。

### 2.8 内容 / 品牌负责人

结论：通过。

通过点：

- 技术方案延续了“项目管理自评诊断报告”的命名。
- CMMI 仍为“参考映射”，没有认证误导。
- 邮件与飞书内容包含免责声明，方向正确。

问题：

- 邮件和飞书卡片中的自动文案需要避免“正式报告”“认证结果”等词。

建议：

- P2：邮件和飞书模板需要在实现前由内容角色过一遍，确保表达专业、克制。

### 2.9 创始人本人

结论：有条件通过。

通过点：

- 能实现“用户填写问卷后，自动留下联系方式并通知我”的核心目标。
- 邮件通知到 `Laura_fan@126.com`，符合早期运营方式。
- 飞书可配置，不影响第一期上线。

问题：

- 如果只完成技术接口，但没有线索导出/查看能力，后续跟进仍然不稳。
- SMTP 授权码、飞书 Webhook、部署环境需要提前准备。

建议：

- P1：上线前确认 `Laura_fan@126.com` 是否支持 SMTP 授权码。
- P1：上线前确认是否本期配置飞书 Webhook。
- P1：必须提供线索导出或人工查询方式。

### 2.10 技术负责人 / 架构

结论：有条件通过。

通过点：

- 架构足够轻量，适合 MVP。
- 前后端边界清晰。
- 安全边界基本合理：密钥只放后端环境变量，日志脱敏。

问题：

- 生产部署形态仍未定，这是最大不确定项。
- 如果静态站点直接用文件方式打开，无法承载后端 API。
- SQLite 是否可用取决于部署环境的持久化能力。

建议：

- P1：确认部署方案为以下之一：
  1. Node.js 服务同时托管静态文件和 API。
  2. 静态站点 + 独立 Node.js API，同域名反向代理 `/api`。
  3. 静态站点 + Serverless API + PostgreSQL。
- P1：技术方案中补充最终推荐路径。若当前没有云环境，建议先用 Node.js 服务托管静态文件和 API，SQLite 存储。
- P1：补充 `.env.example`，避免环境变量遗漏。

## 3. P1 必改项

| 编号 | 问题 | 影响 | 责任方 | 修改建议 |
|---|---|---|---|---|
| P1-01 | 生产部署方式未锁定 | 影响后端、数据库和上线 | 技术负责人 / 后端 | 明确 Node 托管静态文件 + API，或静态站 + API 反代，或 Serverless + PostgreSQL |
| P1-02 | 数据库最终选型未锁定 | 影响实现和数据可靠性 | 后端 / 架构 | 根据部署环境确认 SQLite 或 PostgreSQL |
| P1-03 | 线索人工查看 / 导出缺失 | 售前无法兜底跟进 | 后端 / 创始人 | 增加 CSV 导出脚本或查询说明 |
| P1-04 | SMTP 可用性未确认 | 邮件通知可能无法上线 | 创始人 / 后端 | 确认 126 邮箱 SMTP 授权码或替代邮件服务 |
| P1-05 | 本地联调方式未明确 | 开发效率低，接口路径不稳定 | 前端 / 后端 | 明确同一 Node 服务或本地代理 |
| P1-06 | 通知超时策略未明确 | 用户提交等待过长 | 后端 / 架构 | 邮件和飞书设置超时，失败记录状态 |
| P1-07 | 环境变量清单缺少样例 | 容易漏配密钥和路径 | 后端 / 测试 | 增加 `.env.example` |
| P1-08 | 安全测试项不够明确 | 密钥或敏感日志风险 | 测试 / 后端 | 增加源码密钥扫描和日志脱敏检查 |

## 4. P2 优化项

| 编号 | 建议 | 价值 |
|---|---|---|
| P2-01 | 移动端失败后保留滚动位置 | 提升填写体验 |
| P2-02 | 邮件 / 飞书模板由内容角色复核 | 避免品牌和合规风险 |
| P2-03 | 增加按时间范围导出线索 | 便于运营复盘 |
| P2-04 | 增加响应式截图验收 | 降低 UI 返工 |
| P2-05 | 后续改造异步通知队列 | 降低接口响应耗时 |

## 5. 建议修订后的技术决策

为了降低开发不确定性，建议 V2.0-MVP 技术决策如下：

| 事项 | 推荐决策 |
|---|---|
| 本地 / 早期生产部署 | Node.js 服务同时托管静态文件和 `/api` |
| 数据库 | SQLite，数据库文件放在后端数据目录 |
| 后续扩展 | 若迁移到 Serverless 或正式云部署，改 PostgreSQL |
| 表单临时数据 | `sessionStorage` |
| 邮件 | SMTP，测试环境 `EMAIL_MOCK=true` |
| 飞书 | 可配置，未配置返回 `not_configured` |
| 导出 | 提供 CSV 导出脚本 |
| 通知超时 | 邮件 / 飞书单项 3 秒超时，失败记录状态 |
| 接口超时 | 前端 8-10 秒超时 |

## 6. 最终评审结论

当前技术方案有条件通过。

不建议直接开发完整版本；建议先按 P1 修订技术方案，然后进入 V2.0-MVP 开发。

开发准入条件：

1. 技术方案完成 P1 修订。
2. 明确最终部署方式和数据库。
3. 准备 `.env.example`。
4. 明确 SMTP 是否可用。
5. 明确线索导出方式。

完成以上后，可以进入开发实现。
