# 北京润序科技官网 V2.0 获客闭环设计方案多角色评审记录

版本：V2.0 设计评审版  
日期：2026-07-14  
评审对象：`北京润序科技官网V2.0获客闭环设计方案.md`  
对应 PRD：`北京润序科技官网V2.0获客闭环PRD.md`

## 1. 评审结论

结论：有条件通过，可以进入 V2.0 技术方案阶段；高保真设计前需补齐 4 个 P1。

整体判断：

1. 设计方案已经正确承接 V2.0 获客目标。
2. “先展示基础价值，再留资解锁完整报告”的动线合理。
3. 桌面弹窗、移动端整页卡片的方向可落地。
4. 解锁前 / 解锁后信息差设计成立。
5. 隐私、授权、失败兜底和成功反馈均有基本方案。

需要补齐：

1. 留资弹窗具体信息层级。
2. 解锁前 Top 3 如何“半展示”。
3. 完整报告的信息优先级。
4. 移动端表单长度控制。

## 3. 分角色详细评审

### 3.1 业务产品经理

结论：通过。

认可点：

1. “基础结果免费展示，完整报告留资解锁”符合 B2B 咨询获客逻辑。
2. 表单字段控制在可接受范围内。
3. 复制当前摘要作为低意愿用户兜底，避免强拦截导致完全流失。

建议：

| 优先级 | 建议 | 说明 |
|---|---|---|
| P2 | 在 CTA 附近强化报告价值 | 例如“解锁完整治理建议和下一步诊断重点”。 |

### 3.2 售前

结论：通过。

认可点：

1. 解锁后完整报告包含总分、维度、Top 3 和下一步建议。
2. 项目背景字段对判断线索质量有价值。
3. 成功态说明 1-2 个工作日响应，有助于建立用户预期。

建议：

| 优先级 | 建议 | 说明 |
|---|---|---|
| P2 | 项目背景输入框提示更具体 | 建议提示“当前项目阶段、主要问题、团队规模或交付压力”。 |

### 3.3 产品负责人

结论：有条件通过。

问题：

| 优先级 | 问题 | 修改要求 |
|---|---|---|
| P1 | 解锁前 / 解锁后差异还需更具体 | 明确解锁前 Top 3 只展示维度名和分数，不展示建议正文。 |
| P1 | 完整报告价值表达不足 | 在报告解锁页顶部增加“完整报告包含什么”的短说明。 |

认可点：

1. MVP 设计范围没有扩散到后台。
2. 成功 / 失败 / 重试路径完整。
3. 隐私弹层与授权勾选的设计方向正确。

### 3.4 UI/UX 设计

结论：有条件通过。

问题：

| 优先级 | 问题 | 修改要求 |
|---|---|---|
| P1 | 留资弹窗缺少具体布局层级 | 补充弹窗头部、价值说明、字段区、授权区、按钮区、隐私入口顺序。 |
| P1 | 移动端表单可能过长 | 移动端采用整页卡片，并将项目背景放在最后；按钮固定在卡片底部或表单底部。 |
| P2 | 错误态视觉还可以更具体 | 定义错误边框、错误文字、首个错误聚焦。 |

认可点：

1. 桌面弹窗、移动整页卡片方向合理。
2. 深海蓝视觉体系可以延续 V1.5。
3. 解锁后报告复用现有结果组件，开发成本较低。

### 3.5 前端研发

结论：有条件通过。

问题：

| 优先级 | 问题 | 修改要求 |
|---|---|---|
| P1 | 前端状态缺少命名 | 设计方案需补充状态：`locked`、`form_open`、`submitting`、`unlocked`、`failed`。 |
| P2 | 表单错误状态需要统一 class | 建议后续技术方案定义 `is-invalid`、`is-submitting`、`is-unlocked`。 |

认可点：

1. 表单字段和校验文案清楚。
2. 失败时不丢失用户填写内容，前端可实现。
3. 复制摘要兜底复用 V1.5 能力。

### 3.6 后端研发

结论：通过。

认可点：

1. 设计方案未新增超出 PRD 的后端能力。
2. UI 不展示内部 leadId、邮件或飞书状态，减少后端状态外露。
3. 失败态统一展示，不暴露内部错误细节。

建议：

| 优先级 | 建议 | 说明 |
|---|---|---|
| P2 | 前端失败文案不要区分邮件或飞书内部失败 | 与 PRD 保持一致。 |

### 3.7 测试

结论：有条件通过。

问题：

| 优先级 | 问题 | 修改要求 |
|---|---|---|
| P1 | 设计验收缺少“关闭弹窗后状态” | 增加：关闭留资弹窗后回到问卷结果页，已填写内容是否保留需明确。建议保留。 |
| P2 | 移动端键盘输入场景需补充 | 验收 375px 下输入项目背景时按钮可达。 |

认可点：

1. 主流程验收清楚。
2. 解锁控制可以测试。
3. 授权和隐私弹层可测试。

### 3.8 内容 / 品牌负责人

结论：通过。

认可点：

1. 文案没有夸大诊断报告能力。
2. CMMI 参考映射口径正确。
3. 隐私说明内容完整，能够建立信任。

建议：

| 优先级 | 建议 | 说明 |
|---|---|---|
| P2 | 授权文案要可读，不要使用过浅灰色 | 这是信任信息，不应弱化到不可读。 |

### 3.9 创始人本人

结论：通过。

认可点：

1. 设计能支撑官网从展示型向获客型升级。
2. 不强行拦截所有结果，仍保留复制摘要，比较稳妥。
3. 完整报告解锁能制造合理转化动机。

建议：

| 优先级 | 建议 | 说明 |
|---|---|---|
| P2 | 成功态可以加强“我们会如何联系你” | 例如通过邮箱或飞书联系。 |

### 3.10 技术负责人 / 架构

结论：通过。

认可点：

1. 设计没有要求前端暴露密钥或内部通知状态。
2. 报告解锁状态可由接口返回控制。
3. 设计符合 V2.0-MVP 范围。

建议：

| 优先级 | 建议 | 说明 |
|---|---|---|
| P2 | 高保真前确认状态机 | 避免开发时状态分支遗漏。 |

## 4. 必须修改的问题

### P0

无。

### P1

1. `[P1][解锁差异] 解锁前 / 解锁后差异不够具体 → 影响设计和开发 → 明确解锁前 Top 3 只展示维度名和分数，不展示建议正文。`
2. `[P1][报告价值] 完整报告价值表达不足 → 影响留资转化 → 在表单弹窗和解锁报告顶部说明“完整报告包含治理建议和下一步诊断重点”。`
3. `[P1][弹窗布局] 留资弹窗层级不够具体 → 影响高保真设计 → 明确弹窗顺序：标题、价值说明、字段、授权、隐私入口、按钮。`
4. `[P1][移动端表单] 移动端表单可能过长 → 影响填写体验 → 移动端采用整页卡片，提交按钮放在表单底部且保持可达。`
5. `[P1][前端状态] 状态命名缺失 → 影响开发和测试 → 增加状态：locked、form_open、submitting、unlocked、failed。`
6. `[P1][关闭弹窗] 关闭表单后的状态未定义 → 影响测试 → 关闭后回到问卷结果页，并保留已填写内容。`

### P2

1. 项目背景 placeholder 更具体。
2. 授权文案颜色保证可读。
3. 错误态 class 在技术方案中定义。
4. 移动端键盘输入场景验收。
5. 成功态补充联系方式预期。

## 5. 设计修订建议

建议在设计方案中补充以下内容：

### 5.1 留资弹窗布局

```text
标题：生成完整诊断报告
价值说明：提交咨询信息后，立即查看完整治理建议和下一步诊断重点。
字段区：姓名、公司、联系方式、职位、项目背景
授权区：授权勾选
隐私入口：查看信息使用说明
按钮区：提交并生成报告 / 取消
```

### 5.2 报告解锁状态

```text
locked：基础结果可见，完整建议隐藏
form_open：留资表单打开
submitting：提交中，按钮禁用
unlocked：完整报告可见
failed：提交失败，可重试或复制摘要
```

### 5.3 解锁前 Top 3

```text
显示：维度名称 + 分数
隐藏：完整治理建议
提示：提交咨询信息后查看完整治理建议
```

设计方案有条件通过。

建议下一步：

1. 按 P1 修订设计方案。
2. 再进入 V2.0 技术方案。
3. 技术方案需重点定义前端状态机、接口对接、后端提交和邮件 mock。
