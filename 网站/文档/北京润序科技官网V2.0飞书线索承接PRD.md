## 1. 背景

V2.0 官网问卷已经具备“用户填写问卷 → 留资 → 保存线索 → 解锁项目管理自评诊断报告”的基础闭环。

但从业务运营角度看，仅保存到本地数据库和 CSV 导出还不够。创始人或售前需要在日常工作流里及时看到新线索，并持续记录跟进状态。

当前阶段不建议开发完整官网后台，优先采用飞书承接：

```text
官网问卷提交
→ 后端保存线索
→ 飞书群收到线索卡片
→ 线索写入飞书多维表
→ 创始人 / 售前在飞书内跟进
```

## 2. 产品目标

### 2.1 业务目标

1. 新线索提交后，创始人 / 售前能第一时间在飞书看到。
2. 每条线索自动沉淀到飞书多维表，形成轻量线索池。
3. 支持人工跟进状态管理，避免线索遗漏。
4. 不开发复杂后台，降低 V2.0 交付成本。
5. 为后续判断是否需要 CRM / 后台提供数据基础。

### 2.2 用户目标

用户提交信息后：

1. 页面正常解锁项目管理自评诊断报告。
2. 不需要知道后台使用飞书承接。
3. 不看到飞书发送状态或内部跟进状态。

### 2.3 售前 / 创始人目标

收到线索后，应能快速判断：

1. 这个客户是谁。
2. 是否值得优先跟进。
3. 应该先问什么问题。
4. 当前线索处于什么跟进状态。
5. 是否已预约诊断或转化为服务机会。

## 3. 本期范围

### 3.1 V2.0 本期包含

1. 飞书群机器人线索卡片。
2. 飞书多维表核心字段写入。
3. 线索写入飞书多维表的产品规则。
4. 飞书通知失败与多维表写入失败的处理规则。
5. 线索状态枚举。
6. 售前基础跟进字段。
7. 去重规则。
8. 验收标准。
9. 飞书卡片 mock 与多维表 mock。
10. 失败补录规则。

### 3.2 V2.0-MVP 边界

V2.0-MVP 只做：

1. 飞书群线索卡片。
2. 飞书多维表核心字段新增记录。
3. 本地 SQLite 记录飞书卡片状态和多维表写入状态。
4. 未配置 / 写入失败时不阻断报告解锁。
5. CSV 导出兜底。
6. 失败补录脚本或补录规则。

V2.0-MVP 暂不做：

1. 飞书卡片按钮更新跟进状态。
2. 多维表状态反向同步到本地数据库。
3. 自动记录首次跟进时间。
4. 自动分配跟进负责人。
5. 自动预约日历。

### 3.3 本期不包含

1. 官网后台。
2. 用户登录。
3. 销售线索分配系统。
4. 自动拨打电话或自动添加飞书联系人。
5. 飞书多维表双向同步到官网。
6. 复杂权限管理。
7. 自动日历预约。
8. CRM 集成。

## 4. 数据事实源

V2.0 明确：

```text
本地 SQLite 是线索事实源
飞书群卡片是即时提醒
飞书多维表是运营副本 / 轻量线索池
CSV 导出是兜底查看与补录手段
```

规则：

1. 官网报告解锁只依赖本地 SQLite 保存成功。
2. 飞书卡片失败不影响报告解锁。
3. 多维表写入失败不影响报告解锁。
4. 当本地 SQLite 与飞书多维表不一致时，以本地 SQLite 为准。
5. 可通过 CSV 导出或补录脚本修复飞书多维表缺失记录。

## 5. 核心流程

```text
用户提交官网问卷留资
→ 后端校验通过
→ 后端保存 SQLite 线索
→ 后端生成线索摘要
→ 发送飞书群线索卡片
→ 写入飞书多维表
→ 前端解锁项目管理自评诊断报告
→ 售前在飞书多维表更新跟进状态
```

说明：

1. 线索保存成功仍然是报告解锁条件。
2. 飞书卡片失败或多维表写入失败，不阻断用户查看报告。
3. 飞书失败状态需要记录到本地数据库。
4. 后续可通过失败日志或本地 CSV 补录。

## 6. 飞书线索卡片需求

### 6.1 触发时机

当官网后端完成线索保存后触发飞书卡片。

触发条件：

1. `FEISHU_WEBHOOK_URL` 已配置。
2. 当前环境未开启 `FEISHU_MOCK=true`。
3. 线索保存成功。

未配置飞书时：

```text
notify_feishu=not_configured
```

测试环境：

```text
FEISHU_CARD_MOCK=true
```

开启后不发送真实飞书卡片，但本地记录：

```text
notify_feishu=mock
```

### 6.2 卡片标题

```text
新官网项目诊断线索｜{{风险等级}}｜{{公司}}
```

示例：

```text
新官网项目诊断线索｜极高风险｜衔远科技
```

### 6.3 卡片首屏信息

为保证手机端可快速行动，卡片顶部必须优先展示：

```text
{{风险等级}}｜{{线索等级}}｜{{是否重复}}
公司：{{公司 / 团队}}
姓名：{{姓名}}
联系方式：{{联系方式}}
建议首问：{{建议首问问题}}
```

### 6.4 卡片完整内容结构

卡片优先展示行动判断信息。

```text
【线索摘要】
公司：{{公司 / 团队}}
姓名：{{姓名}}
联系方式：{{联系方式}}
角色：{{职位 / 角色}}
线索等级：{{high / medium / unknown}}
是否重复：{{是 / 否}}

【诊断结果】
项目成功率自评得分：{{总分}}
风险等级：{{风险等级}}
CMMI：{{CMMI 参考映射}}

【Top 3 短板】
1. {{维度}} {{分数}}分
2. {{维度}} {{分数}}分
3. {{维度}} {{分数}}分

【项目背景】
{{项目背景，超长时截断到 120 字，完整内容进入多维表}}

【建议跟进动作】
{{建议跟进动作}}

【建议首问问题】
{{建议首问问题}}
```

### 6.5 卡片按钮

如飞书能力支持，卡片提供按钮：

| 按钮 | 动作 |
|---|---|
| 查看线索表 | 打开飞书多维表 |
| 标记已联系 | 后续版本支持，本期可不做 |
| 复制联系方式 | 飞书客户端内手动复制，本期不做自动按钮 |

V2.0-MVP 只要求“查看线索表”按钮可配置；如果暂不支持按钮，卡片正文放多维表链接。

V2.0-MVP 不做：

1. 标记已联系。
2. 自动更新跟进状态。
3. 复制联系方式按钮。

这些能力放到 V2.1。

### 6.6 卡片展示原则

1. 不出现“官方 CMMI 认证”。
2. 不使用“正式评估报告”。
3. 不承诺一定能解决问题。
4. 保留“项目管理自评诊断”语境。
5. 手机端能在首屏看到公司、姓名、联系方式、分数和建议首问问题。

## 9. 写入多维表规则

### 9.1 写入时机

本地 SQLite 保存成功后，调用飞书多维表写入能力。

### 9.2 成功规则

飞书多维表返回成功后：

```text
bitableStatus=success
```

本地数据库同步记录：

```text
notify_bitable=success
```

### 9.3 失败规则

如果多维表未配置：

```text
notify_bitable=not_configured
```

如果写入失败：

```text
notify_bitable=failed
```

失败不阻断：

1. 页面报告解锁。
2. 本地 SQLite 保存。
3. 邮件通知。
4. 飞书群卡片通知。

### 9.4 去重规则

本期不覆盖历史记录。

同联系方式 10 分钟内重复提交：

1. 本地仍保存一条新记录。
2. 多维表仍新增一条记录。
3. `是否重复=true`。
4. 飞书卡片标题或摘要中标注“重复线索”。

### 9.5 失败补录规则

V2.0-MVP 需要提供补录规则或脚本。

补录对象：

```text
notify_bitable=failed
notify_bitable=not_configured
```

补录方式：

1. 后端读取本地 SQLite 中多维表未成功写入的线索。
2. 重新调用飞书多维表写入。
3. 成功后更新 `notify_bitable=success`。
4. 失败时保留失败状态，并记录失败原因。

建议脚本：

```text
node scripts/retry-bitable.js
```

如果本期暂不开发自动补录脚本，必须支持 CSV 导出后人工导入多维表。

## 10. 状态流转

### 10.1 跟进状态

```text
未跟进
→ 已联系
→ 已预约
→ 已诊断
→ 已转化
```

异常状态：

```text
无效
```

### 10.2 状态说明

| 状态 | 定义 |
|---|---|
| 未跟进 | 线索已进入表格，尚未联系 |
| 已联系 | 已通过电话 / 邮箱 / 飞书联系 |
| 已预约 | 已约定诊断时间 |
| 已诊断 | 已完成首次项目诊断 |
| 已转化 | 已进入付费服务、合作或明确商机 |
| 无效 | 联系不上、非目标客户、无真实需求或重复无效 |

## 11. 权限与安全

### 11.1 配置安全

以下配置只能存在后端环境变量：

1. 飞书机器人 Webhook。
2. 飞书多维表 App Token。
3. 飞书多维表 Table ID。
4. 飞书开放平台凭证。

不得进入前端代码。

### 11.2 数据安全

1. 飞书群建议只加入创始人、售前或实际跟进人。
2. 多维表权限设置为仅指定成员可见。
3. 不在公开群转发客户联系方式。
4. 删除数据时，本地数据库和飞书多维表都需要处理。

### 11.3 权限验收要求

上线前必须确认：

1. 飞书群成员仅包含创始人、售前或实际跟进人。
2. 飞书多维表仅指定成员可访问。
3. 多维表不允许组织内所有人可见。
4. 多维表不允许公开分享链接。
5. Webhook 和多维表 Token 不出现在前端源码、浏览器请求或 Git 提交中。
6. 退出跟进的人员需要移出飞书群和多维表权限。

### 11.4 隐私口径

用户授权文案已经说明信息用于项目诊断沟通。飞书作为内部跟进工具，属于该用途范围。

如果后续接入第三方 CRM 或外部系统，需要更新隐私说明。

## 14. 风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| 飞书 Webhook 未配置 | 收不到卡片 | 本地数据库和 CSV 导出兜底 |
| 多维表写入失败 | 线索池不完整 | 记录失败状态，后续通过补录脚本或 CSV 补录 |
| 飞书权限设置过宽 | 客户信息泄露 | 限制群和多维表成员 |
| 字段过多导致表格难用 | 售前维护困难 | 分核心字段和跟进字段，视图筛选 |
| 跟进状态不维护 | 数据失真 | 固定每日线索检查节奏 |

## 16. 结论

当前阶段不建议做官网运营后台。

V2.0 应优先采用：

```text
飞书线索卡片即时提醒
+ 飞书多维表轻量线索池
+ 邮件备份
+ SQLite / CSV 兜底
```

该方案成本低、上线快、贴合创始人早期亲自跟进线索的业务状态。
