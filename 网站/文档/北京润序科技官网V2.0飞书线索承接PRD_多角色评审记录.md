# 北京润序科技官网 V2.0 飞书线索承接 PRD 多角色评审记录

日期：2026-07-14  
评审对象：`北京润序科技官网V2.0飞书线索承接PRD.md`

评审角色：

- 业务产品经理
- 售前
- 产品负责人
- UI/UX 设计
- 前端研发
- 后端研发
- 测试
- 内容 / 品牌负责人
- 创始人本人
- 技术负责人 / 架构

## 1. 总体结论

结论：有条件通过。

本 PRD 的方向正确：当前不做官网后台，优先用“飞书线索卡片 + 飞书多维表”承接官网线索，符合润序科技当前早期获客阶段。

通过原因：

1. 能解决“页面提交成功但业务侧不知道去哪看”的问题。
2. 不引入复杂后台，交付成本可控。
3. 飞书符合创始人和售前的日常工作流。
4. 多维表可作为轻量 CRM，支持跟进状态维护。
5. 本地 SQLite 和 CSV 仍然保留兜底。

但进入开发前需要修订部分 P1：

1. V2.0-MVP 是否同时做“飞书卡片 + 多维表写入”需要明确优先级。
2. 多维表字段较多，需要区分“首期必填字段”和“后续可选字段”。
3. 飞书开放能力配置项需要在技术方案中明确。
4. 写入失败后的补录方式需要定义。
5. 多维表权限和客户信息安全需要单独验收。

## 2. 分角色评审意见

### 2.1 业务产品经理

结论：通过，有 P1 修订项。

通过点：

- 从业务阶段看，不做后台是正确选择。
- 飞书卡片解决即时提醒，多维表解决持续跟进。
- 状态流转“未跟进 → 已联系 → 已预约 → 已诊断 → 已转化 / 无效”符合售前流程。

问题：

- 字段较多，首期如果全部实现，容易拖慢交付。

建议：

- P1：拆分字段为“首期必填字段”和“增强字段”。
- P1：V2.0-MVP 优先保证卡片通知和多维表核心字段写入。

### 2.2 售前

结论：通过。

通过点：

- 卡片首屏包含联系方式、分数、风险等级、建议首问问题，售前可直接行动。
- 多维表有跟进状态、下次跟进时间、跟进备注，能支撑轻量销售动作。
- 高优先级视图有价值。

问题：

- 如果字段太多，售前维护成本会上升。
- 跟进状态需要简单，不要变成复杂 CRM。

建议：

- P1：多维表首期字段控制在 15 个左右。
- P2：后续再增加“下次跟进时间”“跟进负责人”等字段。

### 2.3 产品负责人

结论：通过。

通过点：

- PRD 范围清楚，不做后台、不做 CRM、不做自动预约。
- 失败不阻断用户报告解锁，与 V2.0 获客闭环主口径一致。
- 多维表作为轻量线索池符合 MVP。

问题：

- V2.0-MVP 与 V2.1 边界还需要更硬。

建议：

- P1：V2.0-MVP 明确只做“卡片 + 多维表新增记录 + 失败状态记录”。
- P2：卡片按钮更新状态放到 V2.1。

### 2.4 UI/UX 设计

结论：通过。

通过点：

- 用户侧体验不受影响，仍然是提交后解锁报告。
- 飞书承接属于后台运营，不需要新增前端页面。
- 卡片信息层级较清晰。

问题：

- 卡片内容较长，手机端可能首屏看不到最关键动作。

建议：

- P1：飞书卡片顶部固定展示“联系方式、风险等级、建议首问问题”。
- P2：项目背景长文本建议截断，完整内容在多维表查看。

### 2.5 前端研发

结论：通过。

通过点：

- 本需求主要是后端和飞书侧，不需要改动用户前端主流程。
- 前端仍只关心保存成功后解锁报告。

问题：

- 如果未来需要展示飞书写入状态，会增加前端复杂度。

建议：

- P1：V2.0 前端不展示飞书卡片或多维表写入状态。
- P2：前端最多展示“我们已收到你的咨询信息”，不暴露内部运营链路。

### 2.6 后端研发

结论：有条件通过。

通过点：

- 飞书 Webhook 卡片实现相对简单。
- 多维表写入可以作为保存后的非阻断动作。
- 本地数据库仍保留最终记录，符合兜底逻辑。

问题：

- 多维表写入不是普通 Webhook，通常需要开放平台凭证、App Token、Table ID、字段 ID 映射。
- PRD 目前没有区分“字段名”和“字段 ID”，实现时需要映射配置。
- 失败补录需要脚本，否则写入失败后只能人工处理。

建议：

- P1：技术方案中增加多维表配置项：`FEISHU_APP_ID`、`FEISHU_APP_SECRET`、`BITABLE_APP_TOKEN`、`BITABLE_TABLE_ID`、字段映射。
- P1：增加失败补录脚本：读取本地 `notify_bitable=failed/not_configured` 的线索并重试写入。
- P2：字段 ID 映射建议放配置文件，不硬编码在业务代码里。

### 2.7 测试

结论：有条件通过。

通过点：

- 飞书卡片验收、多维表验收、安全验收已列出。
- 失败不阻断报告解锁的口径清晰。

问题：

- 缺少 mock 策略：飞书卡片 mock、多维表 mock、写入失败 mock。
- 多维表字段较多，需要明确首期必测字段。

建议：

- P1：补充测试环境 mock：`FEISHU_CARD_MOCK`、`BITABLE_MOCK`。
- P1：首期字段验收只测核心字段。
- P2：补充重复线索写入多维表的测试用例。

### 2.8 内容 / 品牌负责人

结论：通过。

通过点：

- 卡片文案没有使用“官方认证”“正式评估”等风险词。
- “项目管理自评诊断”语境清楚。
- 建议首问问题适合售前开场。

问题：

- 飞书卡片属于内部信息，但仍要避免过强承诺。

建议：

- P2：建议跟进动作统一使用“建议优先确认”，不要使用“必须解决”“立即整改”等强命令表达。

### 2.9 创始人本人

结论：通过。

通过点：

- 飞书是当前最适合的线索承接方式，比后台轻。
- 多维表能让我随时看所有线索，不依赖开发人员查数据库。
- 邮件、数据库、CSV 仍然兜底，风险可控。

问题：

- 不希望字段太多，导致自己不想维护。

建议：

- P1：首期多维表只保留能帮助跟进的字段。
- P2：后续根据真实线索再扩展字段。

### 2.10 技术负责人 / 架构

结论：有条件通过。

通过点：

- 不开发后台的决策正确。
- 飞书卡片和多维表是合理的低成本运营架构。
- 本地 SQLite 仍作为源数据，飞书作为运营视图，职责清楚。

问题：

- 多维表写入涉及外部 API 凭证，不能低估配置成本。
- 如果飞书 API 失败，需要避免阻塞主流程。
- 需要明确“本地数据库是事实源，飞书多维表是运营副本”。

建议：

- P1：明确本地 SQLite 是 source of truth。
- P1：飞书卡片和多维表写入都必须非阻断。
- P1：补充失败补偿脚本。

## 3. P1 必改项

| 编号 | 问题 | 影响 | 责任方 | 修改建议 |
|---|---|---|---|---|
| P1-01 | 多维表字段过多 | 增加实现和维护成本 | 产品 / 售前 | 拆分首期字段和增强字段 |
| P1-02 | MVP 边界不够硬 | 可能拖慢上线 | 产品负责人 | V2.0-MVP 只做卡片、核心字段写入、失败状态 |
| P1-03 | 多维表 API 配置未定义 | 后端无法实现 | 后端 / 架构 | 增加 App ID、Secret、App Token、Table ID、字段映射 |
| P1-04 | 缺少失败补录方案 | 飞书写入失败后数据不完整 | 后端 | 增加补录脚本 |
| P1-05 | 缺少 mock 策略 | 测试环境难验收 | 测试 / 后端 | 增加卡片 mock、多维表 mock |
| P1-06 | 卡片首屏信息需压缩 | 手机端不易行动 | UI/UX / 售前 | 首屏展示联系方式、风险等级、首问问题 |
| P1-07 | 事实源未明确 | 数据一致性争议 | 架构 | 明确 SQLite 是事实源，飞书是运营副本 |
| P1-08 | 权限验收需更具体 | 客户信息泄露风险 | 内容 / 测试 | 增加飞书群和多维表成员权限检查 |

## 4. P2 优化项

| 编号 | 建议 | 价值 |
|---|---|---|
| P2-01 | 卡片按钮更新跟进状态放到 V2.1 | 降低 V2.0 复杂度 |
| P2-02 | 项目背景在卡片中截断，完整内容进表格 | 提升移动端阅读效率 |
| P2-03 | 增加首次跟进时间字段 | 便于计算首响时长 |
| P2-04 | 增加线索来源细分 | 支持后续转化分析 |
| P2-05 | 增加按失败状态重试写入能力 | 降低人工补录 |

## 5. 建议修订后的 V2.0-MVP 范围

### 5.1 必须做

1. 飞书群线索卡片。
2. 飞书多维表核心字段写入。
3. 多维表写入状态记录。
4. 飞书卡片状态记录。
5. 本地 SQLite 作为事实源。
6. CSV 导出继续保留。
7. 飞书配置 mock。
8. 多维表配置 mock。

### 5.2 暂不做

1. 飞书卡片按钮更新状态。
2. 自动修改多维表跟进状态。
3. 自动预约日历。
4. 官网后台。
5. CRM 集成。

### 5.3 首期多维表核心字段

建议 V2.0-MVP 只写入以下字段：

1. 线索ID。
2. 提交时间。
3. 来源。
4. 公司 / 团队。
5. 姓名。
6. 联系方式。
7. 职位 / 角色。
8. 项目背景。
9. 项目成功率自评得分。
10. 风险等级。
11. Top 3 短板。
12. 建议首问问题。
13. 建议跟进动作。
14. 线索等级。
15. 是否重复。
16. 跟进状态。

## 6. 最终评审结论

当前 PRD 有条件通过。

不建议做官网后台。建议继续推进“飞书线索卡片 + 飞书多维表核心字段写入”。

进入开发前需要先按 P1 修订 PRD，再进入技术方案补充。
