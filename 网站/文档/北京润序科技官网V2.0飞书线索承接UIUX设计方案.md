# 北京润序科技官网 V2.0 飞书线索承接 UI/UX 设计方案

版本：V2.0 P1 修订版  
日期：2026-07-14  
设计角色：UI/UX 设计师 / B2B 线索运营体验设计师  
对应 PRD：`北京润序科技官网V2.0飞书线索承接PRD.md`
修订依据：`北京润序科技官网V2.0飞书线索承接UIUX设计方案_多角色评审记录.md`

## 1. 设计结论

本需求的 UI/UX 重点不在官网前台，而在“线索进入飞书后的运营体验”。

设计目标：

```text
官网提交成功
→ 飞书即时提醒
→ 售前首屏判断优先级
→ 多维表沉淀线索
→ 人工维护跟进状态
```

核心设计原则：

1. 飞书卡片首屏必须支持“马上判断是否要跟进”。
2. 多维表字段要少而有效，避免变成难维护的伪后台。
3. 本地 SQLite 是事实源，飞书是运营副本。
4. 失败状态不打扰用户，但要让内部可发现、可补录。
5. 所有文案保持“项目管理自评诊断”语境，不制造认证误导。

## 2. 体验对象

### 2.1 外部用户

用户只感知：

1. 提交咨询信息。
2. 解锁项目管理自评诊断报告。
3. 收到“润序科技会基于评测结果联系你”的提示。

用户不感知：

1. 飞书卡片发送状态。
2. 多维表写入状态。
3. 内部跟进状态。

### 2.2 内部使用者

内部使用者包括创始人和售前。

他们需要：

1. 第一时间知道有新线索。
2. 在手机端快速看到联系方式。
3. 快速判断风险等级和跟进优先级。
4. 能进入多维表维护状态。
5. 能通过视图查看今日新线索和高优先级线索。

## 3. 飞书卡片设计

### 3.1 信息层级

飞书卡片采用“短版首屏 + 完整版展开”的结构：

```text
短版首屏：行动判断
完整版展开：诊断摘要 + 项目背景 + 线索表入口
```

### 3.2 短版首屏：行动判断区

第一屏必须在手机端无需展开即可看到。

```text
新官网自评诊断线索｜极高风险｜衔远科技

极高风险｜medium｜非重复
姓名：范丽丽
联系方式：13146600896
建议首问：当前项目在「目标与范围」上最影响交付结果的问题是什么？
```

设计要求：

1. 标题包含风险等级和公司名。
2. 第一行展示风险等级、线索等级、是否重复。
3. 联系方式必须靠前。
4. 建议首问问题必须出现在首屏。
5. 重复线索使用“重复线索”显著标记。

### 3.3 完整版展开：诊断摘要区

```text
项目成功率自评得分：32
风险等级：极高风险
CMMI：参考映射：CMMI L1

Top 3 短板：
1. 目标与范围 0分
2. 组织与职责 20分
3. 需求与变更 27分
```

设计要求：

1. 分数和风险等级并列展示。
2. Top 3 只展示维度名和分数，避免卡片过长。
3. CMMI 必须写“参考映射”。

### 3.4 完整版展开：背景与入口区

```text
项目背景：
当前项目需求变更频繁，跨团队协作复杂，希望优先诊断交付风险……

建议跟进动作：
建议优先围绕目标与范围、组织与职责、需求与变更进行首次沟通。

查看线索表：
{{BITABLE_VIEW_URL}}
```

设计要求：

1. 项目背景在卡片中最多展示 120 字。
2. 完整项目背景进入多维表。
3. “查看线索表”作为唯一入口。
4. V2.0 不做“标记已联系”按钮。

### 3.5 字段空值与截断规则

| 字段 | 规则 |
|---|---|
| 职位 / 角色为空 | 不展示该行 |
| 项目背景超过 120 字 | 卡片中截断为 120 字，并追加“……” |
| 建议首问为空 | 使用兜底文案：`当前项目最影响交付结果的问题是什么？` |
| 建议跟进动作为空 | 使用兜底文案：`建议先确认项目目标、范围、职责和当前交付风险。` |
| 线索等级为空 | 展示 `unknown` |
| 重复线索为 true | 首屏展示 `重复线索` |
| 多维表链接为空 | 不展示“查看线索表”入口 |

## 4. 飞书多维表设计

### 4.1 表名

```text
润序科技官网线索池
```

### 4.2 字段分组

多维表字段分为 4 组：

| 分组 | 字段 | 用途 |
|---|---|---|
| 线索身份 | 线索ID、提交时间、来源、公司、姓名、联系方式、职位 | 知道是谁 |
| 诊断判断 | 分数、风险等级、Top 3 短板、建议首问、建议跟进动作、线索等级、是否重复 | 判断值不值得跟 |
| 跟进运营 | 跟进状态 | 维护进展 |
| 系统状态 | 多维表写入状态 | 排查同步问题 |

### 4.3 V2.0-MVP 字段展示顺序

字段定义以 PRD 的 V2.0-MVP 核心字段为准。设计只定义多维表中的展示顺序，不改变字段定义。

建议表格从左到右顺序：

1. 跟进状态
2. 线索等级
3. 风险等级
4. 公司 / 团队
5. 姓名
6. 联系方式
7. 项目成功率自评得分
8. 建议首问问题
9. Top 3 短板
10. 项目背景
11. 建议跟进动作
12. 是否重复
13. 提交时间
14. 来源
15. 职位 / 角色
16. 线索ID
17. 多维表写入状态

排序理由：

1. 售前最先看跟进状态和优先级。
2. 联系信息放在前部，减少横向查找。
3. 线索ID、同步状态放后部，主要用于排查。

### 4.4 字段视觉建议

| 字段 | 视觉建议 |
|---|---|
| 跟进状态 | 单选彩色标签 |
| 线索等级 | high 用红 / 橙，medium 用蓝，unknown 用灰 |
| 风险等级 | 极高风险 / 高风险用红橙色 |
| 是否重复 | 复选框或“重复”标签 |
| 多维表写入状态 | success 绿色，failed 红色，mock 灰色 |

## 5. 多维表视图设计

V2.0-MVP 必建视图：

1. 今日新线索。
2. 高优先级线索。
3. 待跟进。

V2.0-MVP 可选视图：

1. 已转化 / 无效。

### 5.1 今日新线索

用途：每天处理新增线索。

筛选：

```text
提交时间 = 今天
```

排序：

```text
提交时间 倒序
```

### 5.2 高优先级线索

用途：优先跟进高价值或高风险线索。

筛选：

```text
线索等级 = high
或 风险等级 = 极高风险 / 高风险
```

排序：

```text
项目成功率自评得分 升序
```

### 5.3 待跟进

用途：日常工作台。

筛选：

```text
跟进状态 = 未跟进 / 已联系
```

排序：

```text
提交时间 倒序
```

### 5.4 已转化 / 无效

V2.0-MVP 可选。

用途：复盘线索质量。

筛选：

```text
跟进状态 = 已转化 / 无效
```

## 6. 跟进状态设计

### 6.1 状态枚举

```text
未跟进
已联系
已预约
已诊断
已转化
无效
```

### 6.2 状态颜色建议

| 状态 | 颜色建议 | 含义 |
|---|---|---|
| 未跟进 | 灰色 | 尚未处理 |
| 已联系 | 蓝色 | 已触达 |
| 已预约 | 紫色 | 已约时间 |
| 已诊断 | 青色 | 已完成首次诊断 |
| 已转化 | 绿色 | 有商业结果 |
| 无效 | 深灰 | 无需继续 |

## 7. 失败状态设计

### 7.1 卡片失败

内部状态：

```text
notify_feishu=failed
```

UI 表现：

1. 不展示给官网用户。
2. 在本地数据库和 CSV 导出中保留。
3. 后续通过补录或人工检查处理。

### 7.2 多维表写入失败

内部状态：

```text
notify_bitable=failed
```

运营处理：

1. 通过补录脚本重试。
2. 或通过 CSV 导出人工补录。

### 7.3 Mock 状态

测试环境：

```text
notify_feishu=mock
notify_bitable=mock
```

设计要求：

1. Mock 状态只用于测试。
2. 不能让业务误以为真实发送成功。
3. CSV 导出中保留 mock 状态。
4. `FEISHU_CARD_MOCK=true` 时不发送真实卡片。
5. `BITABLE_MOCK=true` 时不写入真实多维表。
6. Mock 状态必须在本地数据库或导出文件中可查。

## 8. 权限体验设计

### 8.1 飞书群权限

建议群成员：

1. 创始人本人。
2. 实际售前跟进人。
3. 技术负责人，仅限联调阶段。

不建议：

1. 全员群。
2. 外部群。
3. 无关项目群。

权限确认责任人：

```text
责任人：创始人本人
协助人：技术负责人
检查时机：上线前
```

### 8.2 多维表权限

建议：

1. 仅指定成员可访问。
2. 不开启公开分享链接。
3. 不允许组织内所有人默认可见。
4. 离开项目的人需要移除权限。

权限确认责任人：

```text
责任人：创始人本人
协助人：技术负责人
检查时机：上线前、成员变更时
```

### 8.3 权限检查记录

上线前需要形成最小检查记录：

| 检查项 | 责任人 | 结果 |
|---|---|---|
| 飞书群成员仅包含授权人员 | 创始人本人 | 待确认 |
| 多维表仅指定成员可见 | 创始人本人 | 待确认 |
| 多维表未开启公开分享链接 | 技术负责人 | 待确认 |
| Webhook / Token 不在前端 | 技术负责人 | 待确认 |

## 9. 文案规范

### 9.1 推荐表达

- 项目管理自评诊断
- 项目成功率自评得分
- CMMI 参考映射
- 建议首问问题
- 建议跟进动作

### 9.2 禁止表达

- 官方评估
- CMMI 认证结果
- 正式诊断报告
- 保证解决
- 必须整改

## 10. 设计验收标准

### 10.1 飞书卡片

- [ ] 手机端首屏可见风险等级、线索等级、公司、姓名、联系方式、建议首问问题。
- [ ] 卡片存在短版首屏和完整版展开结构。
- [ ] 重复线索有明显标记。
- [ ] 项目背景超长时截断到 120 字以内。
- [ ] 职位 / 角色为空时不展示空字段。
- [ ] 建议首问和建议跟进动作存在兜底文案。
- [ ] 卡片包含“查看线索表”入口或多维表链接。
- [ ] 卡片不出现“官方认证”“认证结果”等风险词。

### 10.2 多维表

- [ ] 表名为“润序科技官网线索池”。
- [ ] 字段顺序符合设计方案。
- [ ] 跟进状态默认为“未跟进”。
- [ ] high / medium / unknown 视觉可区分。
- [ ] 风险等级视觉可区分。
- [ ] 字段定义与 PRD V2.0-MVP 核心字段一致。
- [ ] 必建今日新线索、高优先级线索、待跟进 3 个视图。
- [ ] 已转化 / 无效视图可选。
- [ ] `FEISHU_CARD_MOCK=true` 时不发送真实卡片，并可查 mock 状态。
- [ ] `BITABLE_MOCK=true` 时不写入真实多维表，并可查 mock 状态。

### 10.3 权限

- [ ] 飞书群成员已确认。
- [ ] 多维表仅指定成员可见。
- [ ] 未开启公开分享链接。

## 11. 结论

V2.0 飞书线索承接设计建议：

```text
飞书卡片 = 即时行动提醒
飞书多维表 = 轻量运营工作台
SQLite = 事实源
CSV = 兜底导出与补录
```

该设计不增加用户前台复杂度，能让创始人和售前在飞书内完成早期线索跟进。
