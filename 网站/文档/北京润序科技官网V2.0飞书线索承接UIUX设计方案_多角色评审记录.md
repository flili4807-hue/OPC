# 北京润序科技官网 V2.0 飞书线索承接 UI/UX 设计方案多角色评审记录

日期：2026-07-14  
评审对象：`北京润序科技官网V2.0飞书线索承接UIUX设计方案.md`  
对应 PRD：`北京润序科技官网V2.0飞书线索承接PRD.md`

评审角色：

- 业务产品经理
- 售前
- 产品负责人
- UI/UX 设计
- 前端研发
- 后端研发
- 测试
- 内容 / 品牌负责人
- 创始人本人
- 技术负责人 / 架构

## 1. 总体结论

结论：有条件通过。

设计方向正确：

1. 不新增官网后台。
2. 不增加前台用户复杂度。
3. 用飞书卡片解决即时提醒。
4. 用多维表解决轻量运营承接。
5. 明确 SQLite 是事实源，飞书是运营副本。

设计方案已经能支撑 V2.0-MVP，但进入技术方案前建议修订部分 P1：

1. 飞书卡片需要明确“短版”和“完整版”两种结构，避免卡片过长。
2. 多维表字段顺序需要和 PRD 核心字段完全一致，避免研发映射歧义。
3. 视图需要标出 MVP 必建和可选。
4. Mock 状态在多维表中是否写入需要更明确。
5. 权限验收需要增加负责人确认动作。

## 2. 分角色评审意见

### 2.1 业务产品经理

结论：通过，有 P1 修订项。

通过点：

- 设计聚焦运营链路，不扩展后台，符合当前业务阶段。
- 卡片首屏先给行动判断，符合线索转化目标。
- 多维表字段顺序按“跟进优先”排列，便于日常处理。

问题：

- 设计方案里字段顺序和 PRD 核心字段略有表达差异，需要统一。

建议：

- P1：多维表字段顺序以 PRD V2.0-MVP 核心字段为准，设计只调整展示顺序，不改变字段定义。

### 2.2 售前

结论：通过。

通过点：

- 手机端首屏包含联系方式和建议首问问题，这对跟进最关键。
- 高优先级视图、待跟进视图能支撑日常售前动作。
- 跟进状态足够简单。

问题：

- 飞书卡片如果展示完整背景、Top 3、建议动作，手机端可能太长。

建议：

- P1：飞书卡片分为短版首屏和完整版展开信息。
- P2：项目背景卡片内截断到 80-120 字，完整内容只在多维表查看。

### 2.3 产品负责人

结论：通过。

通过点：

- 方案和 PRD 的 MVP 边界基本一致。
- V2.1 能力没有混入 V2.0。
- 不做卡片按钮更新状态，范围控制合理。

问题：

- 视图里“已转化 / 无效”对 MVP 有价值，但不是强依赖。

建议：

- P1：MVP 必建视图为“今日新线索”“高优先级线索”“待跟进”；“已转化 / 无效”可选。

### 2.4 UI/UX 设计

结论：通过。

通过点：

- 设计结构清晰：飞书卡片是即时行动提醒，多维表是运营工作台。
- 字段分组合理。
- 权限体验和文案规范已覆盖。

问题：

- 卡片文案可以更结构化，方便后端模板实现。

建议：

- P1：补充卡片模板字段占位符，区分必填、可选和截断字段。

### 2.5 前端研发

结论：通过。

通过点：

- 本设计不要求前端展示飞书或多维表状态。
- 前端主链路不变，仍然保存成功后解锁报告。

问题：

- 无明显前端实现风险。

建议：

- P2：官网成功提示保持现状，不要加入“已发送飞书”等内部状态文案。

### 2.6 后端研发

结论：有条件通过。

通过点：

- 卡片信息层级清楚，后端可模板化。
- 多维表字段分组能转成字段映射配置。
- Mock 和失败状态设计合理。

问题：

- 设计方案未明确多维表字段 ID 映射落点。
- 飞书卡片“短版 / 完整版”需要明确后端最终输出结构。

建议：

- P1：技术方案中增加 `bitable-field-map.json` 或等价配置。
- P1：卡片模板必须明确字段截断规则和空值规则。

### 2.7 测试

结论：有条件通过。

通过点：

- 设计验收标准覆盖飞书卡片、多维表、权限。
- Mock 状态有明确视觉/运营要求。

问题：

- 需要补充 mock 验收：mock 状态下不发送真实卡片、不写真实表，但本地状态可查。
- 权限验收需要有可执行检查项。

建议：

- P1：增加 mock 验收项。
- P1：增加权限检查责任人和检查结果记录。

### 2.8 内容 / 品牌负责人

结论：通过。

通过点：

- 禁止表达已覆盖“官方评估”“认证结果”“保证解决”等风险词。
- “建议首问问题”“建议跟进动作”表达克制。

问题：

- 卡片标题“新官网项目诊断线索”可接受，但建议保留“自评”语境。

建议：

- P2：卡片标题可改为“新官网自评诊断线索｜{{风险等级}}｜{{公司}}”。

### 2.9 创始人本人

结论：通过。

通过点：

- 手机端首屏能看到联系方式和首问问题，符合实际跟进场景。
- 多维表视图清晰，不需要登录官网后台。
- 不做复杂后台，节省时间。

问题：

- 不希望多维表太复杂。

建议：

- P1：MVP 多维表只保留核心字段，增强字段不要提前加。

### 2.10 技术负责人 / 架构

结论：有条件通过。

通过点：

- 设计没有破坏现有 V2.0 获客闭环架构。
- 飞书作为运营副本，SQLite 作为事实源，架构关系正确。
- 失败补录和 CSV 兜底方向正确。

问题：

- 权限和凭证安全需要在技术方案里变成硬性检查。
- 飞书多维表字段 ID 变化时，需要可维护方案。

建议：

- P1：技术方案补充字段映射配置文件和权限检查清单。
- P1：上线前检查必须包含“飞书群成员、多维表访问权限、公开分享关闭”。

## 3. P1 必改项

| 编号 | 问题 | 影响 | 责任方 | 修改建议 |
|---|---|---|---|---|
| P1-01 | 飞书卡片缺少短版 / 完整版明确结构 | 手机端可能过长 | UI/UX / 售前 | 增加短版首屏和完整版展开模板 |
| P1-02 | 多维表字段顺序需和 PRD 对齐 | 后端映射可能歧义 | 产品 / UI | 以 PRD 核心字段为字段定义，设计只定义展示顺序 |
| P1-03 | MVP 必建视图未明确 | 实施成本不清 | 产品负责人 | 必建今日新线索、高优先级线索、待跟进 |
| P1-04 | 卡片字段缺少空值和截断规则 | 模板实现不稳定 | 后端 / UI | 补充项目背景截断、角色为空、重复标记规则 |
| P1-05 | Mock 验收不够细 | 测试环境误判 | 测试 | 增加 mock 不发真实卡片、不写真实表的验收 |
| P1-06 | 权限验收缺少责任人确认 | 客户信息风险 | 创始人 / 测试 | 增加权限检查责任人和确认项 |
| P1-07 | 字段映射维护方式未体现 | 多维表字段变更难维护 | 后端 / 架构 | 技术方案补充字段映射配置 |

## 4. P2 优化项

| 编号 | 建议 | 价值 |
|---|---|---|
| P2-01 | 卡片标题改为“新官网自评诊断线索” | 强化自评语境 |
| P2-02 | 项目背景截断优先 80 字，最多 120 字 | 提升手机端阅读效率 |
| P2-03 | 已转化 / 无效视图作为可选 | 控制 MVP 成本 |
| P2-04 | 后续增加首次跟进时间 | 支持首响指标 |

## 5. 最终评审结论

当前 UI/UX 设计方案有条件通过。

建议先按 P1 修订设计方案，再进入技术方案。

核心判断不变：

```text
飞书卡片解决即时提醒
飞书多维表解决运营跟进
SQLite 保持事实源
CSV 保持兜底
```
