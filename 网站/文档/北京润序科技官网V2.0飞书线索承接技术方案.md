## 1. 技术结论

本方案在现有 V2.0 获客闭环基础上，增加飞书运营承接能力：

```text
官网线索提交
→ SQLite 保存线索
→ 飞书卡片即时提醒
→ 飞书多维表写入运营副本
→ 失败状态写回 SQLite
→ CSV / 补录脚本兜底
```

核心技术原则：

1. SQLite 是事实源。
2. 飞书卡片是即时提醒。
3. 飞书多维表是运营副本。
4. 飞书失败不阻断报告解锁。
5. 多维表失败不阻断报告解锁。
6. Mock 默认开启，避免测试误发真实数据。
7. 飞书卡片与多维表写入支持独立启停。
8. 上线优先级为：SQLite 保存 > 邮件 > 飞书卡片 > 多维表写入。

## 3. 架构方案

```text
Browser
→ POST /api/leads/diagnostic
→ Node.js API
→ SQLite 保存
→ sendFeishuCard()
→ writeBitableRecord()
→ updateNotifyStatus()
→ 返回 reportUnlocked=true
```

状态关系：

| 能力 | 成功状态 | 未配置状态 | Mock 状态 | 失败状态 |
|---|---|---|---|---|
| 飞书卡片 | `success` | `not_configured` | `mock` | `failed` |
| 多维表写入 | `success` | `not_configured` | `mock` | `failed` |

独立启停：

| 能力 | 开关 | 说明 |
|---|---|---|
| 飞书卡片 | `FEISHU_CARD_ENABLED` | false 时直接记录 not_configured |
| 多维表写入 | `BITABLE_ENABLED` | false 时直接记录 not_configured |

## 4. 数据库改造

### 4.1 表字段新增

当前 `diagnostic_leads` 需增加：

| 字段 | 类型 | 说明 |
|---|---|---|
| `notify_bitable` | text | success / failed / not_configured / mock |
| `bitable_record_id` | text | 飞书多维表记录 ID |
| `bitable_error` | text | 写入失败原因，脱敏保存 |
| `feishu_error` | text | 卡片发送失败原因，脱敏保存 |

### 4.2 兼容策略

服务启动时执行轻量 migration：

```sql
ALTER TABLE diagnostic_leads ADD COLUMN notify_bitable TEXT DEFAULT 'not_configured';
ALTER TABLE diagnostic_leads ADD COLUMN bitable_record_id TEXT DEFAULT '';
ALTER TABLE diagnostic_leads ADD COLUMN bitable_error TEXT DEFAULT '';
ALTER TABLE diagnostic_leads ADD COLUMN feishu_error TEXT DEFAULT '';
```

如果字段已存在，忽略错误。

幂等要求：

1. migration 每次服务启动都可安全执行。
2. 新增字段前先检查 `PRAGMA table_info(diagnostic_leads)`。
3. 如果字段已存在，跳过。
4. 如果使用 `ALTER TABLE ADD COLUMN` 报 duplicate column，捕获并忽略。
5. migration 失败不得破坏已有线索数据。

## 6. 飞书卡片技术方案

### 6.0 实现形态

V2.0-MVP 优先使用飞书 Webhook 文本消息或简单卡片，不做复杂交互卡片。

原因：

1. 降低实现复杂度。
2. 保证手机端可读。
3. 避免飞书交互卡片权限和回调配置阻塞上线。

V2.1 再考虑交互卡片和按钮更新跟进状态。

### 6.1 发送入口

```js
await sendFeishuCard(lead)
```

调用时机：

1. SQLite 保存成功后。
2. 邮件通知前后均可，但不阻断主流程。
3. 单次超时 3 秒。

### 6.2 Mock 规则

```text
FEISHU_CARD_MOCK=true
```

结果：

1. 不请求真实 Webhook。
2. 返回 `mock`。
3. SQLite 记录 `notify_feishu=mock`。

### 6.3 未配置规则

如果 `FEISHU_CARD_ENABLED=false` 或 `FEISHU_WEBHOOK_URL` 为空：

```text
notify_feishu=not_configured
```

### 6.4 卡片模板

标题：

```text
新官网自评诊断线索｜{{level}}｜{{company}}
```

短版首屏：

```text
{{level}}｜{{leadQuality}}｜{{duplicateLabel}}
姓名：{{name}}
联系方式：{{contact}}
建议首问：{{firstQuestion}}
```

完整版内容：

```text

风险等级：{{level}}
CMMI：{{cmmiReference}}

Top 3 短板：
{{weaknessLines}}

项目背景：
{{projectBackgroundShort}}

建议跟进动作：
{{followupAction}}

{{BITABLE_VIEW_URL}}
```

### 6.5 字段处理规则

| 字段 | 规则 |
|---|---|
| `role` 为空 | 不展示 |
| `projectBackground` 超过 120 字 | 截断到 120 字并追加“……” |
| `firstQuestion` 为空 | 使用兜底首问 |
| `followupAction` 为空 | 使用兜底动作 |
| `duplicate=true` | 展示“重复线索” |
| `BITABLE_VIEW_URL` 为空 | 不展示查看线索表入口 |

## 7. 飞书多维表技术方案

### 7.0 前置权限

真实写入飞书多维表前，必须准备飞书企业自建应用能力：

1. 已创建飞书企业自建应用。
2. 已获取 `FEISHU_APP_ID`。
3. 已获取 `FEISHU_APP_SECRET`。
4. 应用具备多维表读写权限。
5. 应用已被授权访问目标多维表。
6. 已获取 `BITABLE_APP_TOKEN`。
7. 已获取 `BITABLE_TABLE_ID`。
8. 多维表字段已按 V2.0-MVP 核心字段创建。

如果以上任一项未满足，多维表写入保持 mock 或 not_configured，不影响官网上线。

### 7.1 写入入口

```js
await writeBitableRecord(lead)
```

调用时机：

1. SQLite 保存成功后。
2. 飞书卡片发送前后均可。
3. 单次超时 3 秒。
4. 失败不阻断报告解锁。

### 7.2 Mock 规则

```text
BITABLE_MOCK=true
```

结果：

1. 不请求真实多维表 API。
2. 返回 `mock`。
3. SQLite 记录 `notify_bitable=mock`。

### 7.3 未配置规则

如果 `BITABLE_ENABLED=false`，或缺少任一配置时：

```text
FEISHU_APP_ID
FEISHU_APP_SECRET
BITABLE_APP_TOKEN
BITABLE_TABLE_ID
```

返回：

```text
notify_bitable=not_configured
```

### 7.4 Access Token

调用飞书开放平台：

```text
POST /open-apis/auth/v3/tenant_access_token/internal
```

请求：

```json
{
  "app_id": "{{FEISHU_APP_ID}}",
  "app_secret": "{{FEISHU_APP_SECRET}}"
}
```

响应中获取：

```text
tenant_access_token
```

Token 可在内存中缓存，有效期按飞书返回值处理。

缓存规则：

1. 服务进程内缓存 `tenant_access_token`。
2. 缓存同时记录过期时间。
3. 距离过期不足 5 分钟时重新获取。
4. 获取失败时，本次多维表写入返回 `failed`。
5. 不把 token 写入日志。

### 7.5 写入记录 API

调用飞书多维表新增记录：

```text
POST /open-apis/bitable/v1/apps/{{BITABLE_APP_TOKEN}}/tables/{{BITABLE_TABLE_ID}}/records
```

Header：

```text
Authorization: Bearer {{tenant_access_token}}
Content-Type: application/json
```

Body：

```json
{
  "fields": {
    "线索ID": "{{leadId}}",
    "提交时间": "{{createdAt}}",
    "来源": "官网问卷",

    "线索等级": "{{leadQuality}}",
```

## 9. 失败补录方案

### 9.1 补录脚本

新增：

```text
node scripts/retry-bitable.js
```

参数：

```text
node scripts/retry-bitable.js --dry-run
node scripts/retry-bitable.js --limit 20
node scripts/retry-bitable.js --status failed
node scripts/retry-bitable.js --status not_configured
```

参数规则：

| 参数 | 说明 |
|---|---|
| `--dry-run` | 只打印将补录的 leadId，不调用飞书 API |
| `--limit` | 限制本次最多处理条数，默认 20 |
| `--status` | 指定补录状态，支持 failed / not_configured |

处理对象：

```text
notify_bitable=failed
notify_bitable=not_configured
```

执行逻辑：

1. 读取 SQLite 中未成功写入多维表的线索。
2. 对每条线索重新调用 `writeBitableRecord()`。
3. 成功后更新 `notify_bitable=success` 和 `bitable_record_id`。
4. 失败后保留失败状态并更新 `bitable_error`。
5. 每次执行必须输出处理数量、成功数量、失败数量。
6. 不在日志中输出完整联系方式、完整项目背景或 token。

### 9.2 人工兜底

如果飞书 API 暂不可用：

1. 执行 CSV 导出。
2. 人工导入飞书多维表。
3. 本地仍保留 SQLite 事实源。

## 10. API 主流程调整

当前 `/api/leads/diagnostic` 调整为：

```text
接收请求
→ 服务端校验
→ SQLite 保存线索，notify_feishu=pending，notify_bitable=pending
→ 发送邮件
→ 发送飞书卡片
→ 写入飞书多维表
→ 更新通知状态
→ 返回 reportUnlocked=true
```

返回给前端：

```json
{
  "success": true,
  "leadId": "lead_xxx",
  "reportUnlocked": true,
  "duplicate": false,
  "notifyStatus": {
    "email": "mock",
    "feishu": "mock",
    "bitable": "mock"
  }
}
```

前端仍不展示 `notifyStatus`。

超时规则：

1. 飞书卡片 3 秒超时。
2. 多维表写入 3 秒超时。
3. 任一超时都记录 failed。
4. 超时不影响 `reportUnlocked=true`。

## 10.1 真实发送灰度流程


3. 阶段 4 前确认权限设置。

## 14. 风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| 飞书 API 配置复杂 | 延迟上线 | Mock 默认开启，卡片和多维表可分步启用 |
| 多维表字段名变化 | 写入失败 | 使用字段映射配置 |
| 飞书权限过宽 | 客户信息泄露 | 上线前权限检查 |
| 飞书 API 超时 | 提交等待变长 | 3 秒超时，失败不阻断 |
| 多维表写入失败 | 运营副本缺失 | SQLite 事实源 + CSV / retry 脚本补录 |
