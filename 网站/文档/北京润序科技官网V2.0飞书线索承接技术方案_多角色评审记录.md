## 1. 总体结论

结论：有条件通过。

技术方案整体可行，已经覆盖：

1. 飞书卡片短版 / 完整版模板。
2. 飞书多维表核心字段写入。
3. SQLite 事实源。
4. `FEISHU_CARD_MOCK` 与 `BITABLE_MOCK`。
5. 多维表字段映射配置。
6. 多维表失败补录脚本。
7. 权限检查清单。
8. Mock / 真实联调 / 安全测试。

但进入开发前仍需修订 P1：

1. 飞书卡片能力应优先用 Webhook 文本卡片，复杂交互卡片放后续。
2. 多维表 API 配置需要说明“企业自建应用权限”前置条件。
3. SQLite migration 需要设计成幂等。
4. `notify_bitable` 字段进入导出 CSV。
5. `retry-bitable.js` 需要限制每次补录条数，避免误刷。
6. 真实发送前需要明确灰度开关。

## 2. 分角色评审意见

### 2.1 业务产品经理

结论：通过。

通过点：

- 方案能解决线索进入飞书、进入多维表的问题。
- 不影响用户报告解锁。
- SQLite 作为事实源，避免飞书失败造成线索丢失。

问题：

- 多维表配置失败时，业务需要知道怎么兜底。

建议：

- P1：CSV 导出中增加 `notify_bitable`，便于识别哪些线索未进表。

### 2.2 售前

结论：通过。

通过点：

- 卡片首屏包含联系方式、风险等级、建议首问问题。
- 多维表记录能支持后续跟进。
- 失败不阻断用户，但内部可通过状态识别。

问题：

- 如果真实飞书卡片内容太复杂，手机端阅读会受影响。

建议：

- P1：V2.0-MVP 先用文本卡片或简单卡片，不做复杂交互卡片。

### 2.3 产品负责人

结论：通过。

通过点：

- V2.0-MVP 不做卡片按钮状态更新，范围控制合理。
- Mock 默认开启，降低误发风险。
- 补录脚本和 CSV 兜底符合 MVP。

问题：

- 真实发送切换缺少灰度步骤。

建议：

- P1：增加灰度开关流程：先卡片真实、多维表 mock；再卡片真实、多维表真实。

### 2.4 UI/UX 设计

结论：通过。

通过点：

- 技术方案承接了短版首屏 / 完整版展开。
- 字段空值和截断规则可以实现。
- 设计不要求前端展示内部状态。

问题：

- 需要避免飞书真实卡片格式失控。

建议：

- P2：上线前用真实飞书手机端截图验证首屏。

### 2.5 前端研发

结论：通过。

通过点：

- 前端无需新增交互。
- `/api/leads/diagnostic` 返回结构兼容，前端仍不展示通知状态。

问题：

- 如果后端通知串行执行过慢，前端等待可能变长。

建议：

- P1：后端必须保证飞书卡片和多维表都有 3 秒超时；后续可异步化。

### 2.6 后端研发

结论：有条件通过。

通过点：

- API、字段映射、token 获取、写入记录、补录脚本均有方案。
- SQLite migration 有基本思路。

问题：

- `ALTER TABLE ADD COLUMN` 直接执行会在字段已存在时报错，需要幂等处理。
- 飞书多维表字段类型和 API 入参格式可能与字段名不完全一致。
- token 缓存需要避免每条线索都重新获取。

建议：

- P1：migration 使用字段检查后再新增，或捕获 duplicate column 错误。
- P1：`bitable-field-map.json` 中允许配置字段名或字段 ID，并在技术实现中兼容。
- P1：tenant_access_token 需要内存缓存。

### 2.7 测试

结论：有条件通过。

通过点：

- Mock、未配置、失败、真实联调和安全测试都有覆盖。
- 权限检查清单明确。

问题：

- 补录脚本测试还不够细。
- 真实联调可能误写多条测试数据。

建议：

- P1：补录脚本支持 `--limit` 和 `--dry-run`。
- P1：真实联调使用专门测试表或测试视图，避免污染正式线索池。

### 2.8 内容 / 品牌负责人

结论：通过。

通过点：

- 卡片标题使用“自评诊断线索”，符合品牌与合规。
- CMMI 坚持“参考映射”。
- 没有官方认证、正式评估等风险表达。

问题：

- 卡片正文需要避免太强烈的命令语气。

建议：

- P2：所有建议文案统一用“建议优先确认”，不用“必须整改”。

### 2.9 创始人本人

结论：通过。

通过点：

- 能让我在飞书看到新线索。
- 多维表能作为早期线索池。
- 即使飞书失败，本地数据仍然保留。

问题：

- 上线前需要我确认飞书群成员和多维表权限。

建议：

- P1：权限检查清单需要作为上线前必做项。

### 2.10 技术负责人 / 架构

结论：有条件通过。

通过点：

- 架构保持轻量。
- SQLite 事实源、飞书运营副本的关系正确。
- Mock 默认开启符合安全策略。

问题：

- 飞书开放平台凭证配置是外部依赖，可能拖慢开发联调。
- 多维表字段配置如果不可用，需要允许卡片先上线。

建议：

- P1：卡片通知和多维表写入支持独立启停。
- P1：上线优先级为：SQLite 保存 > 邮件 > 飞书卡片 > 多维表写入。

## 3. P1 必改项

| 编号 | 问题 | 影响 | 责任方 | 修改建议 |
|---|---|---|---|---|
| P1-01 | 飞书卡片复杂度需收敛 | 实现和移动端展示风险 | 后端 / UI | V2.0-MVP 优先文本卡片或简单卡片 |
| P1-02 | 多维表 API 权限前置未写清 | 联调可能被阻塞 | 后端 / 架构 | 补充企业自建应用权限要求 |
| P1-03 | SQLite migration 幂等性不足 | 重启服务可能报错 | 后端 | 字段存在时跳过或忽略 duplicate column |
| P1-04 | CSV 导出缺少 bitable 状态 | 运营无法识别补录对象 | 后端 / 产品 | 导出 `notify_bitable`、`bitable_record_id` |
| P1-05 | 补录脚本缺少安全参数 | 可能误刷大量数据 | 后端 / 测试 | 增加 `--dry-run`、`--limit` |
| P1-06 | 真实发送缺少灰度流程 | 容易误发或污染表格 | 产品 / 测试 | 先真实卡片 + 多维表 mock，再全量真实 |
| P1-07 | token 缓存未明确 | 性能和稳定性问题 | 后端 | tenant_access_token 内存缓存 |
| P1-08 | 卡片与多维表独立启停未明确 | 外部依赖影响上线 | 架构 | 支持两者独立配置 |

## 4. P2 优化项

| 编号 | 建议 | 价值 |
|---|---|---|
| P2-01 | 真实飞书手机端截图验收 | 保证首屏可读 |
| P2-02 | 测试表 / 测试视图隔离 | 避免污染正式线索池 |
| P2-03 | 后续通知异步队列 | 降低接口等待 |
| P2-04 | 补录脚本记录执行日志 | 方便排查 |

## 5. 最终评审结论

当前技术方案有条件通过。

建议先按 P1 修订技术方案，再进入开发。

修订后可按以下优先级开发：

```text
1. SQLite 字段扩展
2. 飞书卡片 mock / 真实发送
3. 多维表 mock
4. 多维表真实写入
5. CSV 导出增强
6. retry-bitable 补录脚本
7. 权限检查清单
```
