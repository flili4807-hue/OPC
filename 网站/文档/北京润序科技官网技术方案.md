## 1. 技术方案结论

本期官网建议采用纯静态站点方案。

技术评审状态：有条件通过。满足案例公开权限确认、部署环境确认、基础测试通过后，可进入正式上线准备。

原因：

1. 当前产品范围是公司品牌展示、服务说明、案例背书、项目成功率自评和联系方式转化。
2. V1 明确不包含后台 CMS、用户登录、在线支付、预约日历、CRM 自动化、AI 客服和复杂表单。
3. 问卷 V1 不保存用户填写结果，仅在浏览器本地即时计算。
4. 纯静态方案上线快、成本低、风险小、回滚简单，符合当前创业阶段官网诉求。

推荐技术栈：

| 层级 | 方案 |
|---|---|
| 页面 | HTML5 |
| 样式 | CSS3，原生响应式布局 |
| 交互 | 原生 JavaScript |
| 数据存储 | V1 不存储用户数据 |
| 资源 | 本地静态图片、SVG favicon |
| 部署 | 静态托管服务 / 对象存储 + CDN / 公司服务器静态目录 |
| 统计 | V1 可不接入；如接入需补隐私说明 |

## 5. 问卷评分逻辑

### 5.1 输入

每题为一个 `select` 控件：

- 默认值：`1`。
- 可选值：`1`、`2`、`3`、`4`、`5`。

### 5.2 计算规则

```text
总题数 = 24
满分 = 24 × 5 = 120
已选分数和 = 所有 select value 的求和
项目成功率得分 = round(已选分数和 / 120 × 100)
```

### 5.3 状态规则

| 状态 | 判断条件 | 页面表现 |
|---|---|---|
| 初始状态 | 24 题均为默认 0 分 | 显示 0 分、CMMI L1、治理建议、咨询 CTA |
| 调整中 | 用户修改任意题目 | 实时更新 0-100 得分、CMMI 映射、治理建议、咨询 CTA |

### 5.4 分数映射

| 分数 | 判断 | CMMI 映射 | 建议 |
|---:|---|---|---|
| 85-100 | 高成功概率 | L4-L5 | 强化数据驱动和组织级最佳实践沉淀 |
| 70-84 | 可控推进 | L3-L4 | 加强风险前置、发布门禁和复盘闭环 |
| 55-69 | 需要治理 | L2-L3 | 优先补齐计划、依赖、风险和质量机制 |
| 40-54 | 高风险 | L1-L2 | 先做项目治理重构再加速交付 |
| 0-39 | 极高风险 | L1 | 先完成目标、范围、组织和质量体系梳理 |

### 5.5 数据处理

V1 不存储、不上传、不持久化用户填写结果。

所有计算只在浏览器本地完成。页面刷新后结果清空。

## 6. 样式与响应式方案

### 6.1 视觉系统

| 变量 | 用途 |
|---|---|
| `--paper: #06101f` | 页面主背景 |
| `--ink: #f5f8ff` | 主文字 |
| `--muted: #95a8c0` | 辅助文字 |
| `--blue: #3978ff` | CTA、重点强调 |
| `--lime: #c9ff57` | 少量数据强调 |
| `--line: rgba(255,255,255,.13)` | 边框和分割线 |

### 6.2 布局策略

- 桌面端使用 `--shell: min(1180px, calc(100vw - 48px))` 控制内容宽度。
- 桌面端关键模块使用 CSS Grid。
- 移动端通过媒体查询切换为单列布局。

### 6.3 断点

| 断点 | 策略 |
|---|---|
| `max-width: 800px` | 移动端单列布局，隐藏中间导航 |
| `801px - 1100px` | 平板端多列降级，数据/服务/方法论改为 2-3 列 |
| `>1100px` | 桌面端完整布局 |

### 6.4 兼容性

`backdrop-filter` 在部分浏览器上可能降级。当前卡片已有半透明背景作为基础 fallback，即使模糊不生效，也能保持可读。

## 8. 可访问性方案

当前策略：

1. 页面使用语义化结构。
2. 主视觉、二维码等图片提供 `alt`。
3. 导航提供 `aria-label`。
4. 问卷题目使用 `label` 包裹文本和控件。
5. CTA 高度不低于 42px。
6. 移动端不应出现横向滚动。
7. 链接、按钮、问卷选择器已提供 `:focus-visible` 焦点态。
8. 支持 `prefers-reduced-motion`，用户偏好减少动态时降低动画和滚动动效。

建议补充：

1. 检查深色背景下的对比度。
2. 使用键盘 Tab 完整走查导航、CTA、问卷选择器。

## 9. 性能方案

### 9.1 当前性能策略

1. 不引入大型前端框架。
2. 不引入第三方 JS。
3. 仅使用 1 张二维码 JPG 和 1 个 SVG favicon。
4. CSS 单文件。
5. JS 只用于问卷即时计算。

### 9.2 性能目标

| 指标 | 目标 |
|---|---:|
| Lighthouse Performance | ≥ 80 |
| Lighthouse Accessibility | ≥ 80 |
| Lighthouse SEO | ≥ 80 |
| 首屏加载 | ≤ 2 秒，需注明测试网络条件 |

## 10. 安全与合规

### 10.1 当前风险较低

原因：

1. 无登录。
2. 无表单提交。
3. 无数据库。
4. 无用户数据上传。
5. 无第三方统计脚本。

### 10.2 仍需注意

| 风险 | 策略 |
|---|---|
| 客户案例与前雇主信息 | 未确认公开授权时必须泛化 |
| CMMI 表述 | 明确为管理诊断，不等同官方 CMMI 认证 |
| 中国大陆部署 | 如部署在中国大陆服务器，需要 ICP 备案 |
| 后续接入统计 | 需补隐私政策和 Cookie/统计说明 |
| 后续接入表单 | 需说明字段、用途、保存周期、删除方式 |

## 11. 部署方案

### 11.1 推荐方式

优先使用静态托管。

可选：

1. 静态托管平台。
2. 对象存储 + CDN。
3. 公司服务器 Nginx 静态目录。

### 11.2 发布内容

发布目录应包含：

```text
index.html
questionnaire.html
questionnaire.js
styles.css
robots.txt
assets/feishu-qr.jpg
assets/favicon.svg
```

### 11.3 回滚方案

1. 每次上线保留上一版静态文件包。
2. 如果发现联系方式错误、敏感案例展示、页面严重错位，立即回滚上一版。
3. 如果使用静态托管平台，优先使用平台版本回滚能力。

## 13. 需求一致性状态

当前产品 PRD、设计 PRD、技术方案和页面实现已完成以下口径同步：

1. 问卷所有题目默认 0 分，打开后直接展示默认诊断结果。
2. 用户调整题目后实时更新项目成功率得分、CMMI 映射、治理建议和咨询 CTA。
3. 主 CTA 使用“项目诊断 / 预约 30 分钟项目诊断”。
4. 涉及未确认公开的高并发和准确率数据使用泛化表达。

后续如新增表单、统计、报告生成或后端服务，需要同步更新产品 PRD、设计方案和技术方案。
